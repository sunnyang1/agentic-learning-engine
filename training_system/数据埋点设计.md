---

## 一、文档说明

### 1.1 目的

本文档定义 MediaStorm 新人培训与能力图谱系统（Agentic Learning v2.0）中所有需要采集的行为数据点，覆盖学员端、Agent 端、系统端三个维度。通过标准化的数据采集框架，为系统效果评估、Agent 策略优化、业务决策提供数据支撑。

### 1.2 设计原则

| 原则 | 说明 |
|------|------|
| **全链路覆盖** | 从学员登录到 Agent 决策、从内容消费到能力升级，每个关键环节均有埋点 |
| **最小侵入** | 埋点采集不影响学员学习体验和 Agent 响应速度 |
| **隐私优先** | 敏感数据（如作品分析、导师评价）脱敏存储，遵循最小权限原则 |
| **实时可用** | 核心埋点近实时（< 1 分钟）进入分析 pipeline，支撑 Agent 实时决策 |
| **可扩展** | 事件 schema 支持动态扩展属性，适应未来新增 Agent 或功能模块 |

### 1.3 术语定义

| 术语 | 定义 |
|------|------|
| **Event（事件）** | 系统内发生的、可被记录的离散行为或状态变化 |
| **Property（属性）** | 描述事件的维度信息，如时间、用户、设备、上下文 |
| **User Profile（用户画像）** | 描述用户静态/半静态属性的维度表 |
| **Session（会话）** | 用户连续活跃的时间段，默认 30 分钟无操作视为会话结束 |
| **Agent Session（Agent 会话）** | Agent 与用户的单次交互周期，从触发到闭环 |

## 二、Event Dictionary（事件字典）

### 2.1 事件分类总览

```
事件体系
├── 用户行为事件（User Action Events）
│   ├── 登录/访问类
│   ├── 内容消费类
│   ├── 交互操作类
│   └── 社交/反馈类
├── Agent 行为事件（Agent Action Events）
│   ├── Agent 生命周期类
│   ├── Agent 决策类
│   ├── Agent 交互类
│   └── Agent 干预类
└── 系统事件（System Events）
    ├── 错误/异常类
    ├── 性能/超时类
    └── 降级/回退类
```

### 6.3 实验设计规范

#### 6.3.1 分流策略

```python
# 分流算法：基于 user_id 的确定性哈希，确保同一用户始终在同一组
def assign_experiment_group(user_id: str, experiment_id: str, groups: list) -> str:
    hash_input = f"{experiment_id}:{user_id}"
    hash_value = int(hashlib.md5(hash_input.encode()).hexdigest(), 16)
    index = hash_value % len(groups)
    return groups[index]

# 分层分流：同一用户可同时参与多个不相关的实验
# 层 1：内容策略实验（EXP-002, EXP-005）
# 层 2：交互策略实验（EXP-001, EXP-007）
# 层 3：预警策略实验（EXP-003, EXP-004）
# 原则：同一层内的实验互斥，不同层的实验正交
```

#### 6.3.2 样本量计算

| 参数 | 说明 | 默认值 |
|------|------|--------|
| 显著性水平（α） | 假阳性概率 | 0.05 |
| 统计功效（1-β） | 真阳性概率 | 0.80 |
| 最小可检测效应（MDE） | 指标相对变化 | 5% |
| 基线值 | 对照组指标值 | 根据历史数据 |

**样本量公式**：
```
n = 2 * (Z_(1-α/2) + Z_(1-β))^2 * σ^2 / (μ1 - μ0)^2
```

**示例**：课程完成率基线 70%，期望检测到 +5% 的提升（即 73.5%），则每组需要约 3,500 样本。
MediaStorm 团队约 100 人，因此单个实验需要覆盖全员并持续 2-4 周以积累足够样本。

#### 6.3.3 实验监控与止损

| 监控项 | 止损条件 | 动作 |
|--------|----------|------|
| 实验组指标严重恶化 | 核心指标下降 > 10% 且持续 3 天 | 自动暂停实验，全部流量切回对照组 |
| 系统错误率飙升 | 实验组错误率 > 5% | 立即暂停实验 |
| 用户投诉激增 | 实验组投诉量 > 对照组 3 倍 | 人工 review，必要时暂停 |
| 样本不均衡 | 实验组/对照组人数比偏离 50:50 超过 10% | 检查分流逻辑，必要时重新分流 |

## 七、数据治理与隐私合规

### 7.1 数据分级

| 级别 | 数据类型 | 访问控制 | 脱敏要求 | 保留期限 |
|------|----------|----------|----------|----------|
| **L1 公开** | 课程信息、案例标题、技能定义 | 全员可读 | 无需脱敏 | 永久 |
| **L2 内部** | 学习进度（聚合）、能力等级分布 | 学员本人 + 导师 + HR + 管理层 | 对外脱敏 | 永久 |
| **L3 敏感** | 个人学习记录、测评成绩、作品链接 | 学员本人 + 导师 + HR | 姓名脱敏为代号 | 离职后 2 年 |
| **L4 机密** | Agent 决策日志中的原始推理、导师评语原文 | 学员本人 + 导师 + 系统管理员 | 严格脱敏 | 离职后 1 年 |

### 7.2 隐私保护措施

| 措施 | 说明 |
|------|------|
| **数据最小化** | Agent 只访问完成任务所需的最小数据集合 |
| **目的限制** | 作品分析数据仅限培训系统使用，不与绩效考核直接挂钩 |
| **去标识化** | 数据分析时使用 user_hash 替代真实姓名；Procedural Memory 去标识化 |
| **访问审计** | 所有对个人敏感数据的访问记录审计日志 |
| **用户可控** | 学员可查看自己的学习数据，可申请导出或删除 |
| **本地化** | 敏感数据存储在国内，不出境 |

### 7.3 数据质量保证

| 检查项 | 方法 | 频率 |
|--------|------|------|
| 事件完整性 | 对比 SDK 上报数 vs 服务端接收数 | 每日 |
| 属性空值率 | 统计核心属性的空值比例 | 每日 |
| 时间戳一致性 | 客户端时间 vs 服务端时间偏差 < 1 分钟 | 每日 |
| 用户 ID 有效性 | 检查是否存在未注册用户的埋点 | 每日 |
| 指标口径一致性 | 同一指标在不同报表中的数值是否一致 | 每周 |
| 数据血缘追踪 | 从报表指标追溯到原始事件 | 每月 |
