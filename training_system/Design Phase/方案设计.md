## 一、总体架构设计

### 1.1 架构设计原则

| 原则 | 说明 |
|------|------|
| **飞书原生优先** | 优先使用飞书已有能力（多维表格/云文档/智能伙伴/应用引擎），减少自研成本 |
| **分层解耦** | Orchestrator / Memory / Agent / Tool 四层独立演进，接口标准化 |
| **Human-in-the-Loop** | Agent 处理80%标准场景，人工保留20%高价值/异常场景的决策权 |
| **渐进式信任** | MVP 阶段 Agent 只做「推荐」不做「决策」，积累3个月数据后逐步开放自主权 |
| **可审计可解释** | 所有 Agent 决策记录日志，每次推荐说明理由 |

## 二、详细模块设计

### 2.1 Orchestrator（调度中枢）

#### 2.1.1 组件定位

整个 Agent 团队的「项目经理」，负责接收事件、查询记忆、决策调度、管理冲突、协调人机交接。

#### 2.1.2 事件源设计

| 事件类型 | 触发源 | 示例 | 优先级 |
|---------|--------|------|--------|
| **学员行为事件** | 学习系统埋点 | 完成课程、页面停留超阈值、提交作业 | P0 |
| **工作流事件** | 飞书/PR/AE | 学员打开新项目、导出视频、提交审片 | P1 |
| **时间事件** | 定时任务 | 每周一早9点、每日晚8点、入职第N天 | P0 |
| **状态变化事件** | Memory 更新 | 能力等级L1→L2、风险评级🟢→🟡、项目完成 | P0 |
| **外部事件** | 导师/HR/管理者 | 导师标记「需要关注」、HR发起试用期评估 | P1 |

#### 2.1.3 决策引擎三层架构

```
┌─────────────────────────────────────────────────────────────┐
│                     Orchestrator 决策引擎                     │
├─────────────────────────────────────────────────────────────┤
│  LAYER 1: 规则层（Rule Engine）                              │
│  ├─ 硬编码业务逻辑，毫秒级响应                                │
│  ├─ 例：「连续3天未登录 → 必须干预」                          │
│  ├─ 例：「卡顿超阈值 → 必须推送参考」                         │
│  └─ 覆盖率：80%简单场景                                       │
├─────────────────────────────────────────────────────────────┤
│  LAYER 2: 策略层（Policy Engine）                            │
│  ├─ 可配置的干预策略，管理员可调参                            │
│  ├─ 例：「优先Coach Agent vs 优先内容Agent」                  │
│  ├─ 例：「疲劳预警阈值：单日学习时长 > 4小时」                │
│  └─ 覆盖率：15%策略选择场景                                   │
├─────────────────────────────────────────────────────────────┤
│  LAYER 3: 模型层（Model Engine）                             │
│  ├─ 轻量预测模型，处理复杂场景                                │
│  ├─ 例：「该学员此刻最可能接受什么形式的帮助？」              │
│  ├─ 例：「综合多个信号，判断优先级最高的干预方式」            │
│  └─ 覆盖率：5%复杂决策场景                                    │
└─────────────────────────────────────────────────────────────┘
```

#### 2.1.4 冲突解决机制

```
【冲突场景示例】

内容Agent 想推送「网感特训课程」（学员网感得分 L1.5）
         ↘
          └─► Orchestrator 冲突仲裁 ◄─┐
         ↗                            │
预测Agent 检测到学员今日已学4小时、情绪指标下降

仲裁逻辑：
  Step 1: 查询 Memory → 学员今日学习时长4h，历史平均2.5h，已超额
  Step 2: 优先级规则 → 预测Agent「疲劳预警」> 内容Agent「内容推送」
  Step 3: 执行计划：
            • 暂不推送新课程
            • CoachAgent 发送关怀消息
            • 记录决策到 Memory
            • 明日早9点定时事件重新触发
```

### 2.2 Memory（学员记忆）— 四层架构

#### 2.2.1 热记忆（Hot Memory）

```
┌─────────────────────────────────────────────────────────────┐
│  HOT MEMORY — 冻结快照（~1,500 tokens）                      │
│  每次 Agent 推理自动注入，保持 LLM 前缀缓存稳定               │
├─────────────────────────────────────────────────────────────┤
│  📋 USER.md（学员画像卡片）~500 tokens                       │
│     • 基础信息：岗位、等级、入职时间、导师                     │
│     • 学习偏好：视频/图文、早上/晚上、短时/长时、自主/推动    │
│     • 沟通风格：内向/外向、直接给答案/苏格拉底式              │
│     • 当前目标：短期目标（如「2周后独立剪项目」）             │
├─────────────────────────────────────────────────────────────┤
│  📋 MEMORY.md（情境记忆卡片）~800 tokens                     │
│     • 当前项目：类型、阶段、deadline、协作人                  │
│     • 能力缺口 Top 3：实时更新的最紧急待提升项                │
│     • 近期关键事件：最近3件对学员有重大影响的事               │
│     • 导师关系：导师是谁、上次辅导时间、待改进点              │
│     • 风险状态：🟢正常 🟡关注 🔴需干预                       │
├─────────────────────────────────────────────────────────────┤
│  【更新机制】Nudge 触发：每30分钟 或 关键事件后               │
│  【写入策略】写入磁盘，下次 Agent 调用时生效（冻结快照）       │
└─────────────────────────────────────────────────────────────┘
```

#### 2.2.2 冷回忆（Cold Recall）

| 属性 | 设计 |
|------|------|
| **存储** | 飞书多维表格 + SQLite FTS5 |
| **内容** | 所有 Agent-学员对话、Agent 决策日志、学员行为埋点 |
| **检索** | 关键词全文搜索 + 语义相似度检索 + 时间范围过滤 |
| **特点** | 不自动注入 Agent 上下文，仅当 Agent 显式调用时检索 |
| **架构原则** | 由架构决定访问，而非 Agent 自行判断 |

#### 2.2.3 程序性记忆（Procedural Memory）

```
【Skill Doc 自增长闭环】

Agent 成功完成复杂干预
        │
        ▼
┌───────────────┐
│ 合成 Skill Doc │ ◄── 记录：触发条件、策略步骤、工具序列、效果数据
└───────┬───────┘
        │
        ▼
【下次类似场景发生】
        │
        ▼
┌───────────────┐
│ Agent 读取     │ ◄── 加载已有 Skill Doc 作为参考
│ Skill Doc      │
└───────┬───────┘
        │
        ▼
┌───────────────┐
│ 执行 → 对比    │ ◄── 按 Skill Doc 执行，对比效果
│ 效果 → 更新    │ ◄── 更新成功率、耗时、改进点
└───────────────┘
```

#### 2.2.4 外部知识（External Knowledge）

| 知识类型 | 存储形式 | 访问方式 | 更新机制 |
|---------|---------|---------|---------|
| LLM Wiki | 语义知识库（基于飞书文档自动索引） | RAG 检索 | 课程/案例更新时自动重新索引 |
| 课程库 | 飞书云文档 + 视频（阿里云OSS） | Tool Registry 查询 | 内容管理员维护 |
| 案例库 | 飞书云文档（结构化 Markdown） | Tool Registry 查询 | 每月更新≥2个 |
| 资源库 | 阿里云OSS + 飞书云文档索引 | Tool Registry 查询 | 版权管理员维护 |
| 导师网络 | 飞书多维表格 | Tool Registry 查询 | HR维护 |
| 项目数据库 | 飞书多维表格 | Tool Registry 查询 | 项目经理维护 |

### 2.4 五大核心 Agent 设计

#### 2.4.1 诊断 Agent（Diagnostic Agent）

```
【工作流】

入职前48h ──► 发送「作品收集链接」（飞书消息）
                  │
                  ▼
Day 1 上午 ──► 分析作品：节奏感、调色水平、字幕规范、叙事逻辑
                  │           └─► LLM 多维度分析 + 规则校验
                  ▼
Day 1 下午 ──► 3道「微场景测试」（非选择题，迷你实操）
                  │   ├─ 测试A：给定30秒素材，按「影视飓风风格」粗剪
                  │   ├─ 测试B：给定3个音乐选项，为指定片段选择最合适配乐
                  │   └─ 测试C：给定一段文案，设计分镜节奏（文字描述）
                  ▼
Day 1 傍晚 ──► 生成「能力缺口热力图」+「个性化新手村路径」
                  │
                  ▼
              写入 Memory + 通知 Orchestrator
```

**技术实现**：飞书问卷收集作品 → 应用引擎调用 LLM API 分析 → 多维表格存储结果

#### 2.4.2 Coach Agent（教练 Agent）

```
【核心能力】

1. 苏格拉底式对话
   输入：学员提问 + Hot Memory（学员画像/能力缺口）+ Cold Recall（历史对话）
   处理：不直接给答案，通过递进式提问引导思考
   输出：反问 → 学员回答 → 再追问 → 总结洞察

2. 实时纠错
   输入：学员提交的剪辑脚本/分镜（飞书文档）
   处理：LLM 分析文本 + 对比品牌规范/最佳实践
   输出：飞书文档批注（具体位置+问题+建议）

3. 引导式复盘
   输入：项目完成事件 + 项目数据（完播率/互动率）+ 导师评价
   处理：LLM 生成启发式问题
   输出：主动发起复盘对话
```

**技术实现**：飞书智能伙伴（Copilot）+ 自建知识库（飞书文档）+ 飞书文档批注 API

#### 2.4.3 内容 Agent（Content Agent）

```
【动态路径组装算法】

输入：
  ├─ user_id
  ├─ 诊断报告 / 最新能力图谱
  ├─ 学员目标（如「2周后独立剪项目」）
  └─ 约束条件：{time, format, project_type, preference}

处理：
  Step 1: 查询内容库（课程/案例/文档/模板）
  Step 2: 过滤：学员当前等级 vs 目标等级 → 跳过已掌握内容
  Step 3: 排序：学习偏好匹配度 + 内容新鲜度 + 项目相关性
  Step 4: 组装：视频课 + 案例拆解 + 微实操 + 苏格拉底对话
  Step 5: 估算总时长，确保在约束范围内

输出：
  ┌──────────────────────────────────────────┐
  │ 个性化学习套餐（JSON）                     │
  │ • 核心目标                                 │
  │ • 内容列表（类型/时长/来源）               │
  │ • 完成标准                                 │
  │ • 预期能力增长                             │
  └──────────────────────────────────────────┘
```

**动态调整规则**：
- 连续3次得分>85% → 自动减少相关课程，增加进阶内容
- 停留时间>平均值200% → 自动推送「简化版」或「1对1真人辅导」预警
- 每周一早9点 → 自动推送本周学习套餐到飞书消息

#### 2.4.4 预测 Agent（Predictive Agent）

```
【风险监控模型】

输入信号：
  ├─ 登录行为：连续未登录天数、消息响应速度
  ├─ 学习行为：模块停留时间、重复观看次数、跳过率
  ├─ 测评数据：得分趋势、错题分布、复测次数
  ├─ 项目数据：提交频率、返工率、deadline 延迟率
  └─ 对话数据：Coach Agent 对话参与度、情绪指标（响应速度/文本长度）

模型：
  ├─ 规则层：硬阈值触发（连续3天未登录、停留时间>200%）
  └─ 模型层：轻量分类器（XGBoost/LightGBM）预测「流失风险」概率

输出：
  ┌──────────────┬──────────┬────────────────────────────┐
  │ 风险信号      │ 阈值     │ 自动干预动作                │
  ├──────────────┼──────────┼────────────────────────────┤
  │ 连续3天未登录 │ 触发     │ 飞书私信+5分钟短视频重启     │
  │ 停留时间>200% │ 触发     │ 推送简化版+询问需1对1辅导    │
  │ 测评连续下降  │ 触发2次  │ 通知导师+HR，标记关注名单    │
  │ 作业质量波动  │ 触发     │ 主动发起复盘对话             │
  │ 入职7天完成<60%│ 触发    │ 升级预警给直属领导           │
  └──────────────┴──────────┴────────────────────────────┘
```

## 四、技术架构

### 4.1 技术选型

| 层级 | 组件 | 方案 | 选型理由 |
|------|------|------|---------|
| **编排层** | Orchestrator | 飞书应用引擎 Workflow + 自建规则引擎 | 飞书原生工作流处理常规分支，复杂决策用自建规则引擎 |
| **编排层** | 事件总线 | 飞书多维表格变更触发 + Webhook | 利用飞书原生事件机制，无需额外中间件 |
| **记忆层** | 结构化记忆 | 飞书多维表格 | 学员状态、能力图谱、Agent决策日志的持久化存储 |
| **记忆层** | 非结构化记忆 | 飞书云文档 + LLM wiki | 对话历史、案例内容的语义检索。LLM wiki由LLM自动索引 |
| **智能体层** | LLM 基座 | 阿里云百炼 / 字节豆包 / Claude API | 国内可用、成本可控、支持Function Calling |
| **智能体层** | Agent 框架 | LangChain / 自建轻量框架 | LangChain的AgentExecutor适合原型，后期可自建简化版 |
| **工具层** | 内容工具 | 飞书云文档 API + 多维表格 API | 原生能力，无需额外开发 |
| **工具层** | 通讯工具 | 飞书机器人 API + 飞书消息 API | 推送、对话、会议创建一站式解决 |
| **工具层** | 项目工具 | PR/AE插件（远期）+ 飞书项目 API | 近期通过飞书文档收集项目信息，远期集成剪辑软件 |
| **数据层** | 视频存储 | 阿里云OSS / 飞书云端 | 大容量、低成本、CDN加速 |

### 4.3 关键技术挑战与解法

| 挑战 | 影响 | 解法 |
|------|------|------|
| **Agent 幻觉（Hallucination）** | 生成错误的学习内容或误导性反馈 | ① RAG强制检索：Agent回答必须基于知识库 ② Human-in-the-Loop：关键决策人工审核 ③ 置信度阈值：低置信度标记为「待核实」 |
| **Agent 协作冲突** | 多个Agent给出矛盾建议 | ① Orchestrator统一仲裁 ② Memory共享确保信息一致 ③ 冲突解决规则：预测Agent > 内容Agent（学员状态优先） |
| **实时性要求** | 实战Agent需要秒级响应 | ① 规则引擎处理80%简单场景（<100ms）② LLM处理20%复杂场景（<3s）③ 预加载常用内容到缓存 |
| **数据隐私** | 学员作品和能力数据敏感 | ① 最小权限原则：Agent只能访问需要的Memory ② 数据脱敏：决策日志隐藏学员真实姓名 ③ 本地化处理：敏感数据不出境 |
| **飞书API限制** | 高频调用可能触发限流 | ① 批量操作替代单条操作 ② 本地缓存减少API调用 ③ 异步队列削峰填谷 |
| **LLM 依赖风险** | 供应商故障导致系统不可用 | ① 设计「降级模式」：规则引擎接管关键功能 ② 多LLM供应商备选 ③ 关键功能不100%依赖LLM |

## 七、风险应对

### 7.1 技术风险

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|---------|
| **Agent 幻觉导致误导** | 中 | 高 | ① RAG强制检索，回答必须基于知识库 ② Human-in-the-Loop关键决策人工审核 ③ 置信度阈值：低置信度标记「待核实」 ④ 建立Agent决策日志审计机制 |
| **Agent 协作冲突** | 中 | 中 | Orchestrator统一仲裁 + Memory共享 + 冲突解决规则库。MVP阶段限制Agent并发数量，逐步扩展 |
| **LLM 供应商故障** | 中 | 高 | 设计「降级模式」：LLM不可用时规则引擎接管关键功能。多供应商备选（百炼/豆包/Claude） |
| **飞书 API 限流** | 低 | 中 | 批量操作替代单条操作；本地缓存减少API调用；异步队列削峰填谷 |
| **系统与飞书整合困难** | 低 | 中 | 优先使用飞书原生能力，Agent层完全基于飞书生态构建 |

### 7.2 业务风险

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|---------|
| **资深员工不愿意分享** | 中 | 高 | 设计激励机制（积分/奖金/荣誉），让分享=收益。Agent辅助降低分享成本：口述→Agent自动整理为结构化案例 |
| **Tim没有时间配合拆解案例** | 高 | 高 | 先让资深剪辑师/编导拆解，Tim的爆款作为「特别篇」。Agent辅助：上传素材和成片，Agent自动提取时间轴、生成初版拆解框架，专家只需审核修正 |
| **新人觉得系统枯燥不用** | 低 | 中 | v2.0从根本上解决：Agent主动推送，学员无需「打开系统找课」。对话式交互天然比「点击课程列表」更有趣 |
| **学员对 Agent 不信任** | 中 | 中 | 透明度设计：每次推荐说明理由。渐进式信任：MVP阶段Agent只做「推荐」不做「决策」 |
| **数据隐私与合规** | 低 | 高 | 最小权限原则 + 数据脱敏 + 本地化处理。学员作品分析数据仅限培训系统，不与绩效直接挂钩 |
