## 二、问题 1：Memory Cold Storage 存储限制

### 2.1 评审结论

**结论：飞书多维表格的JSON字段存在大小限制，大量对话历史存储可能超限。需要增加存储容量评估和数据归档策略。**

### 2.2 详细分析

v2.0将 Cold Storage 设计为：
- 存储：飞书多维表格 + SQLite FTS5
- 内容：所有 Agent-学员对话、Agent 决策日志、学员行为埋点、Skill Doc

**风险 A：飞书多维表格 JSON 字段大小限制**

| 数据类型 | 预估单条大小 | 100人×3个月累计 | 是否超限 |
|---------|------------|----------------|---------|
| Agent-学员对话 | 2-5KB/轮 | ~500MB | ⚠️ 可能超限 |
| 行为埋点 | 0.5KB/条 | ~200MB | ✅ 安全 |
| Agent 决策日志 | 1-3KB/条 | ~150MB | ✅ 安全 |

飞书多维表格的单字段存储限制约为 **500KB-1MB**（具体数值未公开，但实测超过500KB的JSON字段会出现性能问题）。如果一个学员3个月累计对话超过200轮（约1MB），单字段就会超限。

**风险 B：SQLite FTS5 在 Serverless 环境中的可用性**

飞书应用引擎是 Serverless 环境（函数计算），每次调用后实例会被销毁：
- SQLite 是文件型数据库，需要本地文件系统持久化
- Serverless 实例重启后，本地文件会丢失
- 这意味着 FTS5 索引无法持久化，每次都需要重建

**建议修订**：

```
【Memory Cold Storage 修订方案】

LAYER 2: Cold Storage（冷存储）— 分两类存储

┌─────────────────────────────────────────────────────────────┐
│  类型A：结构化数据（小体积）                                │
│  存储：飞书多维表格                                         │
│  内容：Agent决策日志、行为埋点摘要、Skill Doc元数据         │
│  特点：单条<10KB，适合多维表格存储                          │
├─────────────────────────────────────────────────────────────┤
│  类型B：非结构化数据（大体积）                              │
│  存储：飞书云文档（Markdown格式）或阿里云OSS                │
│  内容：完整对话记录、作品分析详情、案例拆解全文             │
│  索引：飞书云文档原生搜索 + 手动维护关键词索引              │
│  特点：无大小限制，支持版本历史                             │
└─────────────────────────────────────────────────────────────┘

【数据归档策略】

- Hot Memory保留最近30天数据
- Cold Storage保留全部历史，但按学员分文档存储
- 超过6个月的对话自动归档到阿里云OSS（压缩存储）
- 归档数据仅用于审计和模型训练，不用于实时检索
```

### 2.3 Action Items

- [ ] **Action 1.1**：验证飞书多维表格JSON字段的实际大小限制（PoC）
- [ ] **Action 1.2**：设计对话数据的飞书云文档存储格式（按学员/按月分文档）
- [ ] **Action 1.3**：制定数据归档策略（6个月自动归档到OSS）

## 三、问题 2：异步队列与"飞书原生优先"原则的冲突

### 3.1 评审结论

**结论：Redis + 常驻Worker的设计需要自建服务层，与"飞书原生优先"原则存在冲突。需要明确自建服务的边界，或寻找飞书原生替代方案。**

### 3.2 详细分析

v2.0在 §4.3.x 中设计了异步队列架构：
- Redis 作为消息队列缓冲
- 4个规则引擎 Worker + 2个 LLM代理 Worker 消费事件

**冲突点**：

| 设计点 | 飞书原生能力 | v2.0方案 | 冲突说明 |
|--------|------------|---------|---------|
| 消息队列 | 无原生消息队列 | Redis自建 | 需要阿里云Redis实例，增加运维负担 |
| 常驻Worker | 飞书应用引擎是事件触发，无常驻Worker | 自建Worker | 需要阿里云ECS/函数计算常驻实例 |
| 缓存服务 | 无原生缓存 | Redis自建 | 同上 |

**"飞书原生优先"原则**（§1.1）明确写道："优先使用飞书已有能力，减少自研成本"。但v2.0的异步队列设计需要：
- 阿里云Redis实例（~200-500元/月）
- 阿里云ECS/函数计算常驻实例（~300-800元/月）
- 额外的运维监控

**替代方案评估**：

| 方案 | 可行性 | 延迟 | 成本 | 建议 |
|------|--------|------|------|------|
| A. 完全自建（v2.0当前） | ✅ 可行 | 低 | 中高 | 作为备选 |
| B. 飞书应用引擎异步Workflow | ⚠️ 部分可行 | 中 | 低 | 优先尝试：利用Workflow的延时节点+条件分支实现简单排队 |
| C. 阿里云函数计算 + 定时触发 | ✅ 可行 | 中 | 低 | 替代常驻Worker：每1分钟触发一次，消费队列中的事件 |
| D. 飞书多维表格自动化 | ⚠️ 部分可行 | 高 | 极低 | 利用多维表格的"记录变更触发自动化"实现简单事件处理 |

**建议修订**：

```
【异步队列分层方案】

第一层：飞书原生能力（优先使用）
├─ 飞书应用引擎 Workflow 延时节点 → 简单排队（延迟5-30秒可接受）
├─ 飞书多维表格自动化 → 记录变更触发处理（适合状态流转）
└─ 飞书待办/日程 → 人工任务的队列管理

第二层：阿里云函数计算（必要时使用）
├─ 函数计算定时触发（每1分钟）→ 消费待处理事件
├─ 函数计算异步调用 → LLM推理不阻塞主流程
└─ 阿里云Redis（如需缓存）→ 用函数计算内置缓存替代

原则：
• 80%事件用飞书原生能力处理
• 20%复杂事件（需LLM推理）用函数计算异步处理
• Redis仅在性能测试后确认需要时才引入
```

### 3.3 Action Items

- [ ] **Action 2.1**：评估飞书应用引擎Workflow延时节点的实际延迟和稳定性（PoC）
- [ ] **Action 2.2**：设计阿里云函数计算异步调用方案（替代常驻Worker）
- [ ] **Action 2.3**：明确自建服务层的边界——仅限"异步LLM推理"和"缓存"，其余用飞书原生

## 四、问题 3：Prompt A/B测试在飞书智能伙伴中的可行性

### 4.1 评审结论

**结论：v2.0 §4.1.x 中的A/B测试设计在飞书智能伙伴（Coach Agent载体）中不可行。需要重新设计评估方案。**

### 4.2 详细分析

v2.0设计的A/B测试框架：
- 同一Agent同时运行两个Prompt版本
- 分流比例：50/50
- 评估指标：思考深度评分、对话轮数、满意度、导师人工介入率

**核心问题：飞书智能伙伴不支持A/B测试**

飞书智能伙伴（Lark Copilot）的工作方式：
1. 管理员在后台配置一个知识库 + 一个系统Prompt
2. 所有用户与同一个智能伙伴交互
3. 没有原生的"多版本并行"或"用户分流"能力

这意味着：
- 无法同时运行 coach_agent_v1.2 和 coach_agent_v1.3
- 无法将学员A路由到v1.2、学员B路由到v1.3
- 切换版本需要手动在后台修改配置，全量生效

**可行的替代评估方案**：

| 方案 | 可行性 | 实施成本 | 评估周期 | 建议 |
|------|--------|---------|---------|------|
| A. 飞书智能伙伴后台手动切换 | ✅ 可行 | 极低 | 1周/版本 | **推荐**：周一切换v1.3，观察一周指标，周五决定是否回滚 |
| B. 自建ChatBot替代飞书智能伙伴 | ⚠️ 可行 | 高 | 实时 | 需要自建服务端+飞书机器人API，失去飞书原生优势 |
| C. 小流量灰度（仅限内部测试群） | ✅ 可行 | 低 | 3-5天 | **推荐**：先在内部测试群验证新Prompt，再全量上线 |
| D. 离线批量评估 | ✅ 可行 | 中 | 1天 | 用历史对话数据+新Prompt批量测试，人工评估输出质量 |

**建议修订**：

```
【Prompt评估方案（修订版）】

阶段1：离线批量评估（每次Prompt变更必做）
├─ 收集50+历史学员问题
├─ 用新Prompt批量生成回答
├─ 导师人工评估：引导质量评分（1-10分）
└─ 新Prompt评分 > 旧Prompt评分 + 0.5分 → 进入阶段2

阶段2：内部测试群验证（3-5天）
├─ 在「Agent测试群」中部署新Prompt
├─ 邀请3-5名内部成员（含导师阿杰）测试
├─ 收集反馈：是否有引导偏差/幻觉/不当建议
└─ 无严重问题 → 进入阶段3

阶段3：全量上线 + 周级监控（1周）
├─ 周一在飞书智能伙伴后台切换新Prompt
├─ 监控核心指标：
│   ├─ 学员思考深度评分（目标不下降）
│   ├─ 导师人工介入率（目标不上升）
│   └─ 学员满意度（目标不下降）
└─ 周五Review：指标下降>10% → 立即回滚

取消"50/50分流"设计，改为"周级全量切换+回滚"机制
```

### 4.3 Action Items

- [ ] **Action 3.1**：删除§4.1.x中"50/50分流"和"同时运行两个版本"的设计
- [ ] **Action 3.2**：建立50+历史学员问题测试集（用于离线批量评估）
- [ ] **Action 3.3**：建立"周级全量切换+回滚"的SOP文档

## 五、问题 4：内容库建设缺失

### 5.1 评审结论

**结论：v2.0中没有为内容库建设（课程/案例/资源库）分配工期和资源，但内容Agent和诊断Agent严重依赖内容库。这是一个严重的计划盲区。**

### 5.2 详细分析

v2.0 §6.1 实施复杂度评估表中，**没有"内容库建设"这一项**。但：
- 内容Agent需要"查询内容库（课程/案例/文档/模板）"来组装学习路径
- 诊断Agent需要标杆案例来对比学员作品
- AI Assessment Generator需要课程内容来生成测试题
- Coach Agent需要案例拆解作为苏格拉底对话的素材

**当前内容库状态预估**（MediaStorm实际情况）：
- 课程：可能有零散的培训材料，但未系统化
- 案例：Tim的爆款视频有，但缺乏6维度结构化拆解
- 资源：品牌资产可能有，但版权音乐库可能不完整
- 模板：PR/AE模板可能有，但未统一管理和索引

**内容库建设工作量估算**：

| 内容类型 | 当前状态 | 建设工作量 | MVP最低要求 |
|---------|---------|-----------|-----------|
| 品牌规范课程 | 🟡 部分有 | 1周 | 必须有 |
| 达芬奇基础操作 | 🟡 部分有 | 1周 | 必须有 |
| 网感/节奏专题 | 🔴 无系统课程 | 2-3周 | 必须有 |
| 案例拆解（10个） | 🔴 无结构化拆解 | 3-4周 | 至少3个 |
| 资源库（字体/音乐/模板） | 🟡 部分有 | 1-2周 | 基础资源 |
| **总计** | | **8-11周** | **4-5周** |

**关键问题**：内容库建设需要**资深剪辑师/编导**投入时间，但他们是系统的主要用户（导师端），在系统开发期间他们还需要正常工作。

**建议修订**：

```
【内容库建设计划（新增）】

并行于系统开发的Phase 0（Month 0，即正式启动前1个月）：

Week 1-2 ► 内容盘点
├─ 收集现有培训材料、案例视频、品牌资产
├─ 评估缺口：哪些内容已有？哪些需要新建？
└─ 输出：《内容缺口清单》

Week 3-4 ► MVP内容生产（最小可用集合）
├─ 品牌规范课程（1个短视频 + 1个检查清单）
├─ 达芬奇基础速通（1个短视频 + 1个模板）
├─ 网感入门专题（1个短视频 + 1个案例拆解）
├─ 3个标杆案例的结构化拆解（选题/脚本/拍摄/剪辑/包装/数据）
└─ 基础资源包（标准字体 + 版权音乐清单 + PR模板）

内容生产方式：
├─ 资深员工口述 → Agent辅助整理为结构化文档（节省50%时间）
├─ 已有视频素材 → 飞书妙记自动转录 + Agent提取知识点
└─ 外部优质内容 → 购买版权或获得授权

内容Owner：
├─ 总负责人：内容运营（1人，全职投入Phase 0）
├─ 课程审核：阿杰（导师代表，每周4小时）
└─ 案例拆解：各项目编导（按项目轮流，每人1个/月）
```

### 5.3 Action Items

- [ ] **Action 4.1**：新增Phase 0（Month 0）内容库建设计划到 §6.2
- [ ] **Action 4.2**：明确内容Owner和分工（内容运营全职 + 导师兼职审核）

## 六、问题 5：Phase 3 并行风险

### 6.1 评审结论

**结论：Phase 3（Month 5-6）将实战Agent Phase 2 + Procedural Memory + External Knowledge 三个高复杂度任务并行推进，资源存在严重冲突。建议拆分为两个子阶段。**

### 6.2 详细分析

v2.0 §6.2 Phase 3 排期：

```
Month 5-6 [Phase 3: 深度集成]
目标：实战Agent + Memory扩展，实现「高自主式AI」
自主决策比例：70%

Week 1-4 ► 实战Agent Phase 2（PR/AE插件）
Week 3-4 ► Procedural Memory（Skill Doc）
Week 5-6 ► External Knowledge（LLM Wiki + RAG）
Week 7-8 ► 全量上线（全岗位100人）
```

**资源冲突分析**：

| 任务 | 复杂度 | 关键资源 | 与谁冲突 |
|------|--------|---------|---------|
| 实战Agent Phase 2 | 🔴 高 | 需要PR/AE插件开发工程师 | 后端开发 |
| Procedural Memory | 🟡 中 | 需要LLM Prompt工程师 | Prompt调优 |
| External Knowledge | 🟡 中 | 需要向量数据库+检索工程师 | 后端开发 |
| 全量上线 | 🟡 中 | 需要全员参与测试和培训 | 所有人 |

问题：
1. **实战Agent Phase 2** 需要视频编辑软件（PR/AE）的SDK开发经验，这个技能在团队中可能没有
2. **Procedural Memory** 需要3个月的数据积累才能启动，但Phase 3开始时（Month 5）才刚积累完数据
3. **全量上线**在Week 7-8，但前面3个并行任务如果延期，会直接压缩测试时间

**建议修订**：

```
【Phase 3 拆分方案】

Month 5   [Phase 3a: Memory扩展]
目标：Procedural Memory + External Knowledge上线
自主决策比例：60%→70%

Week 1-2 ► Procedural Memory（Skill Doc）
         ► 基于3个月积累的数据，自动合成第一批Skill Doc
         ► 人工审核后发布

Week 3-4 ► External Knowledge（LLM Wiki + RAG）
         ► 课程/案例语义索引上线
         ► Agent回答质量提升（从"规则匹配"到"语义检索"）

Week 5-6 ► 灰度测试（30人）
         ► 验证Memory扩展后的Agent效果
         ► 收集反馈，修复问题

Month 6   [Phase 3b: 实战Agent Phase 2]
目标：PR/AE插件或飞书侧边栏集成
自主决策比例：70%

Week 1-4 ► 实战Agent Phase 2技术预研和开发
         ► 如果PR/AE插件技术门槛过高，改为飞书侧边栏方案
         ► 如果两者都不可行，保持MVP的飞书消息模式，优化推荐精准度

Week 5-6 ► 全量上线准备
         ► 全员培训（导师/学员/管理层）
         ► 数据迁移（如有遗留数据）
         ► 监控告警配置

Week 7-8 ► 全量上线（100人）
         ► 分批次：剪辑师→编导→其他岗位
         ► 上线后第一周全团队on-call
```

### 6.3 Action Items

- [ ] **Action 5.1**：拆分Phase 3为3a（Memory扩展）和3b（实战Agent Phase 2）
- [ ] **Action 5.2**：实战Agent Phase 2增加技术预研环节（Week 1评估PR/AE插件可行性）
- [ ] **Action 5.3**：全量上线增加分批次策略（剪辑师→编导→其他）

## 七、问题 6：成本估算缺失

### 7.1 评审结论

## 八、问题 7：MVP 人工介入比例

### 8.1 评审结论

**结论：v2.0 MVP流程图中几乎每个环节都有【人工确认】，实际自主决策比例可能远低于目标的20%。需要重新评估并增加自动化程度。**

### 8.2 详细分析

v2.0 §6.3 MVP 核心闭环：

```
学员入职 ──► 诊断Agent分析作品 ──► 【人工确认】诊断结果
                                    │
                                    ▼
              内容Agent推荐路径 ──► 【人工确认】路径
                                    │
                                    ▼
              CoachAgent推送首日任务 ──► 学员在飞书完成
                                    │
                                    ▼
                              【人工审阅】完成状态
```

**问题**：3个【人工确认/审阅】节点意味着：
- 导师需要确认诊断结果（每次新人入职）
- 导师需要确认学习路径（每次路径调整）
- 导师需要审阅完成状态（每次任务完成）

对于1名导师带3名学员、每周6个任务的节奏：
- 每周人工确认次数 = 3名学员 × 6任务 × 3节点 = **54次**
- 每次确认假设2分钟 = **108分钟/周**
- 加上其他工作，导师每周在系统上的时间可能超过3小时

这与v2.0的目标"导师基础答疑时间 30%→15%"矛盾——如果系统本身增加了54次/周的人工确认，导师的时间节省会被抵消。

**建议修订**：

```
【MVP自动化程度分层】

Tier 1: 全自动（无需人工确认）
├─ 每日学习套餐推送（Content Agent自动组装）
├─ Coach Agent苏格拉底对话（全自动）
├─ 行为埋点自动记录（全自动）
├─ 能力图谱自动更新（基于规则自动计算）
└─ 预测Agent规则预警（全自动，仅通知导师）

Tier 2: 半自动（人工抽检，非全量确认）
├─ 诊断Agent作品分析 → 系统自动生成结果，导师抽检20%
├─ 学习路径推荐 → 系统自动推送，导师仅在"异常"时介入
└─ 任务完成状态 → 系统自动标记，导师抽检10%

Tier 3: 人工确认（必须人工审核）
├─ 能力图谱正式发布（影响晋升）
├─ 学员风险升级（🟡→🔴）
└─ 导师对Agent结果的异议申诉

【修订后的MVP闭环】

学员入职 ──► 诊断Agent分析作品 ──► 自动生成结果
                                    │
                                    ▼（导师抽检20%，发现问题可修正）
              内容Agent推荐路径 ──► 自动推送
                                    │
                                    ▼（导师仅在"异常"时介入）
              CoachAgent推送任务 ──► 学员在飞书完成
                                    │
                                    ▼（自动标记完成）
                              能力图谱自动更新
                                    │
                                    ▼（月度导师Review时人工确认）
                              发布正式能力评估
```

### 8.3 Action Items

- [ ] **Action 7.1**：重新设计MVP流程图，区分Tier1/Tier2/Tier3自动化层级
- [ ] **Action 7.2**：定义"异常"触发条件（如：诊断结果与导师判断偏差>2个等级）
- [ ] **Action 7.3**：增加导师抽检界面设计（批量审核+一键确认）

## 九、问题 8：团队飞书应用引擎学习曲线

### 9.1 评审结论

**结论：v2.0的28周工期中没有为团队熟悉飞书应用引擎预留学习时间。团队可能需要2-4周来掌握飞书开发，这会影响实际交付。**

### 9.2 详细分析

v2.0假设团队从Day 1就能高效使用飞书应用引擎，但实际上：
- 飞书应用引擎是低代码平台，但复杂逻辑（条件分支、循环、API调用）仍需学习
- 飞书智能伙伴的Prompt配置有特定的格式和限制
- 飞书多维表格的自动化流程与常规编程思维不同
- 团队可能没有人有飞书开发经验

**学习曲线估算**：

| 技能 | 学习周期 | 影响 |
|------|---------|------|
| 飞书应用引擎Workflow基础 | 3-5天 | 可并行于需求分析 |
| 飞书应用引擎复杂逻辑（分支/循环/API） | 1-2周 | 影响Orchestrator开发 |
| 飞书智能伙伴配置 | 2-3天 | 影响Coach Agent开发 |
| 飞书多维表格自动化 | 2-3天 | 影响Memory和Tool Registry |
| 飞书机器人API | 1-2天 | 影响消息推送 |

**总学习周期：2-3周**

**建议修订**：

```
【Phase 0（Month -0.5）：技术熟悉期】

在正式启动开发前，预留2周让技术团队熟悉飞书生态：

Week 1: 飞书基础能力上手
├─ Day 1-2: 飞书应用引擎Workflow基础（Hello World + 简单分支）
├─ Day 3-4: 飞书多维表格API读写 + 自动化流程
└─ Day 5: 飞书机器人消息推送

Week 2: 飞书进阶能力 + PoC验证
├─ Day 1-2: 飞书应用引擎调用外部API（百炼LLM）
├─ Day 3-4: 飞书智能伙伴配置 + 知识库接入
└─ Day 5: 团队PoC Demo（一个简单的Agent对话闭环）

产出：《飞书开发规范文档》+ 可复用的Workflow模板
```
