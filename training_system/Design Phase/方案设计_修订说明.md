## 修订 1：Memory 架构简化（影响章节 §2.2）

### 原设计
Memory 四层架构：Hot / Cold Recall / Procedural / External

### 修订后
**MVP阶段（Phase 1-2）：两层架构**

```
┌─────────────────────────────────────────────────────────────┐
│  MVP MEMORY ARCHITECTURE（两层）                            │
├─────────────────────────────────────────────────────────────┤
│  LAYER 1: Hot Memory（热记忆）                              │
│  ├─ 存储：飞书多维表格 JSON 字段                            │
│  ├─ 内容：USER.md（学员画像）+ MEMORY.md（情境记忆）        │
│  ├─ 体积：~1,500 tokens（保持不变）                         │
│  ├─ 更新：每30分钟 Nudge 或关键事件后更新                   │
│  └─ 用途：每次 Agent 推理自动注入上下文                      │
├─────────────────────────────────────────────────────────────┤
│  LAYER 2: Cold Storage（冷存储）                            │
│  ├─ 存储：飞书多维表格 + SQLite FTS5                        │
│  ├─ 内容：所有 Agent-学员对话、决策日志、行为埋点、Skill Doc │
│  ├─ 检索：关键词全文搜索 + 时间范围过滤                     │
│  └─ 用途：按需检索，不自动注入上下文                         │
└─────────────────────────────────────────────────────────────┘
```

**Phase 3+ 扩展为四层：**
- Procedural Memory（Skill Doc 自增长）：Phase 3 引入
- External Knowledge（LLM Wiki + RAG）：Phase 3 引入

### 修订理由
- 四层架构在低代码平台实现成本过高
- Procedural Memory 的"自增长自改进"需要大量数据积累（至少3个月）
- External Knowledge 的 RAG 检索在 MVP 阶段可用飞书云文档原生搜索替代

## 修订 2：性能指标调整（影响章节 §4.1、§4.3）

### 原设计

| 指标 | 原目标 |
|------|--------|
| Orchestrator 决策延迟 | <3s |
| 规则引擎响应 | <100ms |
| Agent 消息响应 | <5s |

### 修订后

| 指标 | 修订目标 | 说明 |
|------|---------|------|
| **规则引擎决策延迟** | **<2s（80%场景）** | 含飞书Webhook触发延迟500ms-2s |
| **LLM辅助决策延迟** | **<5s（20%场景）** | 复杂场景异步处理 |
| **规则引擎执行** | **<200ms** | 自建规则引擎内部执行时间 |
| Agent 消息响应 | <5s（保持不变） | |
| 数据查询 | <1s（保持不变） | |

### 新增设计：异步决策模式

```
【复杂场景异步决策流程】

学员事件触发
    │
    ▼
┌─────────────────┐
│ Orchestrator    │ ◄── 立即返回"已收到，处理中"（<500ms）
│ 接收事件        │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 异步决策队列    │ ◄── Redis消息队列
│ (Redis)         │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ LLM推理 + 决策  │ ◄── 完成后推送结果
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 飞书消息推送    │ ◄── "处理完成：推荐你..."
└─────────────────┘
```

## 修订 3：增加异步队列与削峰设计（新增章节 §4.3.x）

### 新增内容

```
### 4.3.x 异步队列与限流设计

【削峰填谷架构】

学员行为事件（高峰期50人并发）
        │
        ▼
┌───────────────┐
│  事件网关      │ ◄── 合并同类事件，去重
│  (飞书Webhook) │
└───────┬───────┘
        │
        ▼
┌───────────────┐
│  Redis队列    │ ◄── 按优先级排队
│  (异步缓冲)   │
└───────┬───────┘
        │
        ▼
┌───────────────┐     ┌───────────────┐
│  规则引擎     │     │  LLM代理      │
│  Worker (4个) │     │  Worker (2个) │
│  消费80%事件  │     │  消费20%事件  │
└───────────────┘     └───────────────┘

【限流策略】

| 资源 | 限流策略 | 阈值 |
|------|---------|------|
| 飞书多维表格API | 批量写入 + 本地缓存 | 每5秒批量写入一次 |
| 阿里云百炼API | 令牌桶 + 本地缓存 | 不超过RPM限制的80% |
| 飞书消息推送 | 速率限制 + 排队 | 每秒最多5条消息 |
| 飞书Webhook接收 | 并发控制 | 最大20个并发连接 |

【缓存策略】

| 缓存类型 | 存储 | TTL | 命中场景 |
|---------|------|-----|---------|
| 学习套餐缓存 | Redis | 30分钟 | 相同学员+相同目标 |
| 能力图谱缓存 | Redis | 15分钟 | 频繁查询 |
| 课程搜索结果 | Redis | 1小时 | 热门关键词 |
| LLM响应缓存 | Redis | 30分钟 | 相同问题+相同上下文 |
```

## 修订 4：Prompt工程资源与版本管理（影响章节 §6.1、新增章节）

### 原设计
- Coach Agent 预估工期：3周
- 预留2周Prompt迭代时间

### 修订后
- Coach Agent 预估工期：**5周**（3周开发 + 4周Prompt调优，部分并行）
- **预留4周Prompt迭代时间**
- 配备**专门Prompt工程师**1人

### 新增：Prompt版本管理规范（建议插入 §4.1 后）

```
### 4.1.x Prompt版本管理规范

【命名规则】
{agent_name}_v{major}.{minor}_{date}
例：coach_agent_v1.3_20260420

【版本控制】
- 使用Git管理Prompt文件
- 每次修改提交commit message，格式：
  [agent] vX.Y: {修改原因} | {测试效果变化}

【A/B测试框架】
- 同一Agent同时运行两个Prompt版本
- 分流比例：50/50（小流量测试用10/90）
- 评估指标：
  ├─ 学员思考深度评分（目标 >7/10）
  ├─ 对话轮数（目标 3-8轮）
  ├─ 学员满意度（目标 >4.0/5）
  └─ 导师人工介入率（目标 <20%）

【回滚机制】
- 新版本上线后24小时内监控核心指标
- 指标下降 >10% 时自动回滚到上一个版本
- 回滚时间 <5分钟

【审计日志】
- 记录：Prompt版本号、上线时间、测试数据、上线决策人
- 存储：飞书多维表格，永久保留
```

## 修订 5：AI Assessment Generator 分层目标（影响章节 §2.5、§5.1）

### 原设计
- 无题型分层，统一Owner通过率 >70%
- Owner审阅时间 <10分钟
- 5种题型统一处理

### 修订后

**按题型分层准确率目标：**

| 题型 | 修订目标 | AI生成方式 | 说明 |
|------|---------|-----------|------|
| 概念理解 | ✅ **>75%** Owner通过 | AI全生成 | 学术研究支持概念题准确率80%+ |
| 场景应用 | ⚠️ **>65%** Owner通过 | AI全生成+自检 | 场景题易出歧义 |
| 对比分析 | ⚠️ **>60%** Owner通过 | AI全生成+自检 | 对比题涉及主观判断 |
| 实操任务 | 🔴 **人工出题** | AI辅助（出初稿） | 需要素材和评分标准，AI难以生成 |
| 苏格拉底追问 | 🔴 **人工出题** | AI辅助（出初稿） | 需要对话上下文，生成质量不稳定 |

**Owner审阅时间分阶段目标：**

| 阶段 | 时间目标 | 理由 |
|------|---------|------|
| 第1-3个月 | <15分钟/门课 | 初期Owner需熟悉AI出题风格 |
| 3个月后 | <10分钟/门课 | AI质量提升+Owner经验积累 |

**AI自检增强：**

在原自检（答案依据/难度分布/LO覆盖）基础上增加：
1. **正确答案唯一性**：确保只有一个正确选项
2. **干扰项合理性**：错误选项需"似是而非"
3. **难度标签校验**：根据词汇复杂度/推理步骤自动标记
4. **文化敏感性**：检查敏感话题或文化偏见

## 修订 6：实战Agent简化（影响章节 §2.4.5、§6.1）

### 原设计
- 实战Agent嵌入工作流（PR/AE/飞书侧边栏）
- 预估工期：6周
- 风险等级：🔴 高

### 修订后
- **MVP阶段**：实战Agent简化为"飞书消息推送参考"
  - 学员在飞书中提交项目信息
  - Agent检测到项目类型后，推送相关参考案例和技巧
  - 不追求真正的"工作流嵌入"
- **Phase 2（Month 4+）**：PR/AE插件或飞书侧边栏深度集成
- **预估工期调整**：MVP 2周，Phase 2 6周

### 修订理由
- PR/AE插件开发需要视频编辑软件的SDK，技术门槛高
- 飞书侧边栏深度集成需要飞书开放平台的高级权限
- MVP阶段用飞书消息替代，核心价值（即时指导）不受影响

## 修订 8：风险应对更新（影响章节 §6.3）

### 新增风险

| 风险 | 概率 | 影响 | 应对策略 |
|------|------|------|---------|
| **飞书API限流导致系统不可用** | 中 | 高 | 异步队列削峰 + 本地缓存 + 批量写入 |
| **Prompt调优时间超预期** | 高 | 中 | 预留4周专项时间 + 配备专门Prompt工程师 |
| **LLM推理延迟导致用户体验差** | 中 | 中 | 异步决策模式 + 默认使用快速模型（qwen-turbo） |
| **AI Assessment概念题准确率不达标** | 低 | 中 | 按题型分层设目标 + 人工兜底 |

### 更新原有风险

| 原风险 | 更新 |
|--------|------|
| 实战Agent（高） | 降为MVP中风险（飞书消息替代），Phase 2恢复高风险 |
| Prompt调优（中） | 升为高风险，增加专项资源 |
