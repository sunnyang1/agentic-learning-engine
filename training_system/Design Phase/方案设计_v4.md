## 一、总体架构设计

### 1.1 架构设计原则

| 原则 | 说明 |
|------|------|
| **飞书原生优先** | 优先使用飞书已有能力（多维表格/云文档/智能伙伴/应用引擎），减少自研成本 |
| **分层解耦** | Orchestrator / Memory / Agent / Tool 四层独立演进，接口标准化 |
| **Human-in-the-Loop** | Agent 处理80%标准场景，人工保留20%高价值/异常场景的决策权 |
| **渐进式信任** | MVP 阶段 Agent 只做「推荐」不做「决策」，积累3个月数据后逐步开放自主权 |
| **可审计可解释** | 所有 Agent 决策记录日志，每次推荐说明理由 |

## 二、详细模块设计

### 2.1 Orchestrator（调度中枢）

#### 2.1.1 组件定位

整个 Agent 团队的「项目经理」，负责接收事件、查询记忆、决策调度、管理冲突、协调人机交接。

#### 2.1.2 事件源设计

| 事件类型 | 触发源 | 示例 | 优先级 |
|---------|--------|------|--------|
| **学员行为事件** | 学习系统埋点 | 完成课程、页面停留超阈值、提交作业 | P0 |
| **工作流事件** | 飞书/PR/AE | 学员打开新项目、导出视频、提交审片 | P1 |
| **时间事件** | 定时任务 | 每周一早9点、每日晚8点、入职第N天 | P0 |
| **状态变化事件** | Memory 更新 | 能力等级L1→L2、风险评级🟢→🟡、项目完成 | P0 |
| **外部事件** | 导师/HR/管理者 | 导师标记「需要关注」、HR发起试用期评估 | P1 |

#### 2.1.3 决策引擎三层架构

```
┌─────────────────────────────────────────────────────────────┐
│                     Orchestrator 决策引擎                     │
├─────────────────────────────────────────────────────────────┤
│  LAYER 1: 规则层（Rule Engine）                              │
│  ├─ 硬编码业务逻辑，毫秒级响应                                │
│  ├─ 例：「连续3天未登录 → 必须干预」                          │
│  ├─ 例：「卡顿超阈值 → 必须推送参考」                         │
│  └─ 覆盖率：80%简单场景                                       │
├─────────────────────────────────────────────────────────────┤
│  LAYER 2: 策略层（Policy Engine）                            │
│  ├─ 可配置的干预策略，管理员可调参                            │
│  ├─ 例：「优先Coach Agent vs 优先内容Agent」                  │
│  ├─ 例：「疲劳预警阈值：单日学习时长 > 4小时」                │
│  └─ 覆盖率：15%策略选择场景                                   │
├─────────────────────────────────────────────────────────────┤
│  LAYER 3: 模型层（Model Engine）                             │
│  ├─ 轻量预测模型，处理复杂场景                                │
│  ├─ 例：「该学员此刻最可能接受什么形式的帮助？」              │
│  ├─ 例：「综合多个信号，判断优先级最高的干预方式」            │
│  └─ 覆盖率：5%复杂决策场景                                    │
└─────────────────────────────────────────────────────────────┘
```

#### 2.1.4 冲突解决机制

```
【冲突场景示例】

内容Agent 想推送「网感特训课程」（学员网感得分 L1.5）
         ↘
          └─► Orchestrator 冲突仲裁 ◄─┐
         ↗                            │
预测Agent 检测到学员今日已学4小时、情绪指标下降

仲裁逻辑：
  Step 1: 查询 Memory → 学员今日学习时长4h，历史平均2.5h，已超额
  Step 2: 优先级规则 → 预测Agent「疲劳预警」> 内容Agent「内容推送」
  Step 3: 执行计划：
            • 暂不推送新课程
            • CoachAgent 发送关怀消息
            • 记录决策到 Memory
            • 明日早9点定时事件重新触发
```

### 2.2 Memory（学员记忆）— 两层架构（MVP）

#### 2.2.1 热记忆（Hot Memory）

```
┌─────────────────────────────────────────────────────────────┐
│  HOT MEMORY — 冻结快照（~1,500 tokens）                      │
│  每次 Agent 推理自动注入，保持 LLM 前缀缓存稳定               │
├─────────────────────────────────────────────────────────────┤
│  📋 USER.md（学员画像卡片）~500 tokens                       │
│     • 基础信息：岗位、等级、入职时间、导师                     │
│     • 学习偏好：视频/图文、早上/晚上、短时/长时、自主/推动    │
│     • 沟通风格：内向/外向、直接给答案/苏格拉底式              │
│     • 当前目标：短期目标（如「2周后独立剪项目」）             │
├─────────────────────────────────────────────────────────────┤
│  📋 MEMORY.md（情境记忆卡片）~800 tokens                     │
│     • 当前项目：类型、阶段、deadline、协作人                  │
│     • 能力缺口 Top 3：实时更新的最紧急待提升项                │
│     • 近期关键事件：最近3件对学员有重大影响的事               │
│     • 导师关系：导师是谁、上次辅导时间、待改进点              │
│     • 风险状态：🟢正常 🟡关注 🔴需干预                       │
├─────────────────────────────────────────────────────────────┤
│  【更新机制】Nudge 触发：每30分钟 或 关键事件后               │
│  【写入策略】写入磁盘，下次 Agent 调用时生效（冻结快照）       │
└─────────────────────────────────────────────────────────────┘
```

#### 2.2.2 冷存储（Cold Storage）— 分层存储

【分层设计】

```
┌─────────────────────────────────────────────────────────────┐
│  类型A：结构化数据（小体积）                                │
│  存储：飞书多维表格                                         │
│  内容：Agent决策日志、行为埋点摘要、Skill Doc元数据         │
│  特点：单条<10KB，适合多维表格存储                          │
├─────────────────────────────────────────────────────────────┤
│  类型B：非结构化数据（大体积）                              │
│  存储：飞书云文档（Markdown格式）                           │
│  内容：完整对话记录、作品分析详情、案例拆解全文             │
│  索引：飞书云文档原生搜索 + 手动维护关键词索引              │
│  特点：无大小限制，支持版本历史                             │
├─────────────────────────────────────────────────────────────┤
│  类型C：归档数据（历史数据）                                │
│  存储：阿里云OSS（压缩存储）                                │
│  内容：超过6个月的对话记录、过期决策日志                    │
│  特点：仅用于审计和模型训练，不用于实时检索                 │
└─────────────────────────────────────────────────────────────┘
```

【数据归档策略】

- Hot Memory保留最近30天数据
- Cold Storage保留全部历史，但按学员分文档存储
- 超过6个月的对话自动归档到阿里云OSS（压缩存储）
- 归档数据仅用于审计和模型训练，不用于实时检索

#### 2.2.3 Phase 3 扩展：程序性记忆（Procedural Memory）

> **启用阶段**：Phase 3（Month 5+）
> **前提条件**：积累至少3个月干预数据

        ▼

        │
        ▼
【下次类似场景发生】
        │
        ▼
┌───────────────┐

#### 2.2.4 Phase 1 轻量版 + Phase 3 增强：外部知识（External Knowledge）

> **启用阶段**：Phase 1（Month 1-2）启用轻量版，Phase 3（Month 5+）增强为完整RAG
> **轻量版方案**：飞书云文档原生搜索API + 飞书多维表格关键词索引

**MVP轻量RAG方案（Phase 1）**：

```
【轻量RAG调用链路】

内容Agent / Coach Agent 调用
        │
        ▼
┌─────────────────┐
│ 飞书多维表格索引 │ ◄── 课程库/案例库/资源库的关键词索引（<200ms）
│ 关键词预构建     │     字段：keyword → content_id → storage_location
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  命中索引？      │
│  ├─ 是 → 直接返回相关文档地址
│  └─ 否 → 降级到飞书云文档原生搜索API
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ LLM 生成最终回答 │ ◄── 注入检索到的内容作为上下文
└─────────────────┘
```

- **索引构建**：将课程库/案例库/资源库的关键词预先构建到飞书多维表格中，Agent调用时先查索引（<200ms），再调LLM
- **索引维护**：内容更新时同步更新多维表格关键词索引，由内容Owner负责
- **Phase 3增强**：升级为LLM Wiki语义知识库，支持语义检索（非仅关键词匹配）

**轻量RAG延迟预算（Phase 1）**：

| 路径 | 目标命中率 | 延迟构成 | 内部延迟 | 端到端延迟 |
|------|-----------|---------|---------|-----------|
| 关键词索引命中 | **>70%** | 多维表格查询<200ms + LLM生成<3s | **<3.2s** | **<5s** |
| 降级到云文档搜索 | **<30%** | 云文档API 500ms-2s + LLM生成<3s | **<5s** | **<7s**（⚠️ 可能超标，需优化索引覆盖率） |

> **优化方向**：Phase 0 PoC中验证关键词索引命中率，若<70%则扩充关键词库或调整索引策略。

| 知识类型 | 存储形式 | 访问方式 | 更新机制 | Phase 1可用性 | Phase 3增强 |
|---------|---------|---------|---------|:------------:|:-----------:|

#### 2.4.1 诊断 Agent（Diagnostic Agent）

**技术实现**：飞书问卷收集作品 → 应用引擎调用 LLM API 分析 → 多维表格存储结果

#### 2.4.2 Coach Agent（教练 Agent）

```
【核心能力】

1. 苏格拉底式对话
   输入：学员提问 + Hot Memory（学员画像/能力缺口）+ Cold Recall（历史对话）
   处理：不直接给答案，通过递进式提问引导思考
   输出：反问 → 学员回答 → 再追问 → 总结洞察

2. 实时纠错
   输入：学员提交的剪辑脚本/分镜（飞书文档）
   处理：LLM 分析文本 + 对比品牌规范/最佳实践
   输出：飞书文档批注（具体位置+问题+建议）

3. 引导式复盘
   输入：项目完成事件 + 项目数据（完播率/互动率）+ 导师评价
   处理：LLM 生成启发式问题
   输出：主动发起复盘对话
```

**技术实现**：飞书智能伙伴（Copilot）+ 自建知识库（飞书文档）+ 飞书文档批注 API

#### 2.4.3 内容 Agent（Content Agent）

```
【动态路径组装算法】

输入：
  ├─ user_id
  ├─ 诊断报告 / 最新能力图谱
  ├─ 学员目标（如「2周后独立剪项目」）
  └─ 约束条件：{time, format, project_type, preference}

处理：
  Step 1: 查询内容库（课程/案例/文档/模板）
  Step 2: 过滤：学员当前等级 vs 目标等级 → 跳过已掌握内容
  Step 3: 排序：学习偏好匹配度 + 内容新鲜度 + 项目相关性
  Step 4: 组装：视频课 + 案例拆解 + 微实操 + 苏格拉底对话
  Step 5: 估算总时长，确保在约束范围内

输出：
  ┌──────────────────────────────────────────┐
  │ 个性化学习套餐（JSON）                     │
  │ • 核心目标                                 │
  │ • 内容列表（类型/时长/来源）               │
  │ • 完成标准                                 │
  │ • 预期能力增长                             │
  └──────────────────────────────────────────┘
```

**路径完成率目标：>80%（与需求KPI一致）**

**动态调整规则**：
- 连续3次得分>85% → 自动减少相关课程，增加进阶内容
- 停留时间>平均值200% → 自动推送「简化版」或「1对1真人辅导」预警
- 连续2次未完成路径 → 自动缩短路径长度（减少50%内容）
- 完成率>80%连续2周 → 标记为「高自律」，减少干预频率
- 每周一早9点 → 自动推送本周学习套餐到飞书消息
- 路径完成正向反馈：飞书消息发送「恭喜完成本周路径！」+积分/徽章

#### 2.4.4 预测 Agent（Predictive Agent）

```
【风险监控模型】

输入信号：
  ├─ 登录行为：连续未登录天数、消息响应速度
  ├─ 学习行为：模块停留时间、重复观看次数、跳过率
  ├─ 测评数据：得分趋势、错题分布、复测次数
  ├─ 项目数据：提交频率、返工率、deadline 延迟率
  └─ 对话数据：Coach Agent 对话参与度、情绪指标（响应速度/文本长度）

模型：
  ├─ 规则层（MVP，Phase 1）：硬编码阈值触发（连续3天未登录、停留时间>200%、入职7天完成<60%）
  │   └─ **预期命中率：45-55%（基线）**，存在误报/漏报，仅作为早期预警
  └─ 模型层（Phase 2）：轻量分类器（XGBoost/LightGBM，基于**至少4个月、100+样本**的标注数据训练）预测「流失风险」概率
      └─ **目标命中率：≥60%**

输出：
  ┌──────────────┬──────────┬────────────────────────────┐
  │ 风险信号      │ 阈值     │ 自动干预动作                │
  ├──────────────┼──────────┼────────────────────────────┤
  │ 连续3天未登录 │ 触发     │ 飞书私信+5分钟短视频重启     │
  │ 停留时间>200% │ 触发     │ 推送简化版+询问需1对1辅导    │
  │ 测评连续下降  │ 触发2次  │ 通知导师+HR，标记关注名单    │
  │ 作业质量波动  │ 触发     │ 主动发起复盘对话             │
  │ 入职7天完成<60%│ 触发    │ 升级预警给直属领导           │
  │ 单日学习>4小时│ 触发     │ Coach Agent 关怀消息        │
  └──────────────┴──────────┴────────────────────────────┘
```

### 2.5 AI Assessment Generator（测试题生成器）

**按题型分层准确率目标：**

| 题型 | 目标 | AI生成方式 | 说明 |
|------|------|-----------|------|
| 概念理解 | ✅ **>75%** Owner通过 | AI全生成 | 学术研究支持概念题准确率80%+ |
| 场景应用 | ⚠️ **>65%** Owner通过 | AI全生成+自检 | 场景题易出歧义 |

| 苏格拉底追问 | 🔴 **人工出题** | AI辅助（出初稿） | 需要对话上下文，生成质量不稳定 |

**Owner审阅时间分阶段目标：**

| 阶段 | 时间目标 | 理由 |
|------|---------|------|
| 第1-3个月 | <15分钟/门课 | 初期Owner需熟悉AI出题风格 |
| 3个月后 | <10分钟/门课 | AI质量提升+Owner经验积累 |

### 2.6 季度360度评估（Quarterly 360° Review）

**认证流程**：

Agent自动评估 ──► 导师推荐 ──► 季度Review委员会审核 ──► 飞书公告
      │               │                  │                  │
      ▼               ▼                  ▼                  ▼
  能力图谱达标    导师填写推荐语      HR+资深导师组成      全公司可见
  触发晋升提醒    确认带教记录        委员会季度集中审核    颁发电子徽章
```

**Agent自动评估触发条件**（新增状态校验，防止误触发）：
```
Agent自动评估触发条件：
  ├─ 能力雷达图达标 ✓
  ├─ 项目经历数量达标 ✓
  ├─ 带教记录达标 ✓（L3/L4）
  └─ 【新增】能力图谱状态 = "已发布" ✓
     └─ 若状态 ≠ "已发布"，仅发送"预备晋升提醒"给导师，不创建晋升申请
```

**特权体系**：

| 特权 | L1 | L2 | L3 | L4 |
|------|----|----|----|----|
| 独立承接项目 | ❌ | ✅ | ✅ | ✅ |

| 案例拆解发布 | ❌ | ✅ | ✅ | ✅ |
| 培训预算申请 | ❌ | ❌ | ✅ | ✅ |

**技术实现**：
- 等级状态存储在User表level字段
- 认证记录存储在LevelCertification表（飞书多维表格）
- 晋升审批流通过飞书审批实现：学员申请→导师推荐→能力图谱自动校验→HR审核→飞书公告

#### 性能指标与决策延迟（两层指标）

| 指标 | 目标 | 说明 |
|------|------|------|

| **规则引擎端到端（含Webhook）** | **<2s（80%场景）** | 含飞书Webhook触发延迟500ms-2s，这是飞书生态的固有延迟 |

| 数据查询 | <1s（保持不变） | |

### 4.1.x Prompt版本管理规范

【命名规则】
{agent_name}_v{major}.{minor}_{date}
例：coach_agent_v1.3_20260420

【版本控制】
- 使用Git管理Prompt文件
- 每次修改提交commit message，格式：
  [agent] vX.Y: {修改原因} | {测试效果变化}

【Prompt评估方案（三阶段）】

> **说明**：飞书智能伙伴不支持同时运行多个Prompt版本，因此A/B测试框架改为"离线评估→内部验证→周级全量切换"机制。

**阶段1：离线批量评估（每次Prompt变更必做）**
- 收集50+历史学员问题作为测试集
- 用新Prompt批量生成回答
- 导师人工评估：引导质量评分（1-10分）
- 新Prompt评分 > 旧Prompt评分 + 0.5分 → 进入阶段2

**阶段2：内部测试群验证（3-5天）**
- 在「Agent测试群」中部署新Prompt
- 邀请3-5名内部成员（含导师代表）测试
- 收集反馈：是否有引导偏差/幻觉/不当建议
- 无严重问题 → 进入阶段3

**阶段3：周级全量上线 + 监控（1周）**
- 周一在飞书智能伙伴后台切换新Prompt
- 监控核心指标：
  ├─ 学员思考深度评分（目标不下降）
  ├─ 导师人工介入率（目标不上升）
  └─ 学员满意度（目标不下降）
- 周五Review：指标下降>10% → 立即回滚

【回滚机制】
- 新版本上线后一周内持续监控核心指标
- 指标下降 >10% 或收到严重负面反馈时回滚
- 回滚方式：在飞书智能伙伴后台切换回上一个版本
- 回滚时间 <30分钟（含验证）

【审计日志】
- 记录：Prompt版本号、上线时间、测试数据、上线决策人
- 存储：飞书多维表格，永久保留

### 4.2 飞书生态集成矩阵

```

────────────────────────────────────────────────────────────────

              ──► Memory 结构化数据（学员档案/能力图谱）

### 4.3 关键技术挑战与解法

| 挑战 | 影响 | 解法 |
|------|------|------|
| **Agent 幻觉（Hallucination）** | 生成错误的学习内容或误导性反馈 | ① RAG强制检索：Agent回答必须基于知识库 ② Human-in-the-Loop：关键决策人工审核 ③ 置信度阈值：低置信度标记为「待核实」 |
| **Agent 协作冲突** | 多个Agent给出矛盾建议 | ① Orchestrator统一仲裁 ② Memory共享确保信息一致 ③ 冲突解决规则：预测Agent > 内容Agent（学员状态优先） |
| **实时性要求** | 实战Agent需要秒级响应 | ① 规则引擎内部执行<200ms ② 规则引擎端到端<2s（80%场景，含Webhook延迟500ms-2s）③ LLM推理内部<3s ④ LLM推理端到端<5s（20%场景，异步模式下用户感知<3s）⑤ 预加载常用内容到缓存 |
| **数据隐私** | 学员作品和能力数据敏感 | ① 最小权限原则：Agent只能访问需要的Memory ② 数据脱敏：决策日志隐藏学员真实姓名 ③ 本地化处理：敏感数据不出境 |
| **飞书API限制** | 高频调用可能触发限流 | ① 批量操作替代单条操作 ② 本地缓存减少API调用 ③ 异步队列削峰填谷 |
| **LLM 依赖风险** | 供应商故障导致系统不可用 | ① 设计「降级模式」：规则引擎接管关键功能 ② 多LLM供应商备选 ③ 关键功能不100%依赖LLM |

            Week 7-8  ► 灰度上线（剪辑师15人）

Month 4-5   [Phase 2扩展：认证与激励体系]
            目标：等级认证 + 导师激励上线，与Phase 2能力扩展并行推进
            自主决策比例：50%（不变）
            
            Week 1-2  ► **【新增】等级认证体系**（晋升审批流 + 能力图谱自动校验 + 电子徽章 + 特权配置）
            Week 2-3  ► **【新增】导师激励体系**（积分自动化统计 + 排行榜 + 兑换审批流）
            Week 3-4  ► 认证/激励/评估三个模块联调 + 与Agent闭环集成测试
            Week 5-6  ► 小范围内测（导师5人 + 学员10人）

Month 5     [Phase 3a: Memory扩展]
            目标：Procedural Memory + External Knowledge上线
            自主决策比例：60%→70%
            
            Week 1-2 ► **Procedural Memory（Skill Doc：触发条件库 + 策略步骤库 + 成功率追踪表）**
            Week 3-4 ► External Knowledge（LLM Wiki + 语义RAG检索）
            Week 5-6 ► 灰度测试（30人）

Month 6     [Phase 3b: 实战Agent深度集成]
            目标：实战Agent Phase 2 + 全量上线
            自主决策比例：70%
            
            Week 1-4 ► 实战Agent Phase 2技术预研和开发
            Week 5-6 ► 全量上线准备（培训+监控）
            Week 7-8 ► 全量上线（分批次：剪辑师→编导→其他）

Month 7-12  [Phase 4: 持续优化]
            目标：预测Agent轻量模型 + Agent市场
            自主决策比例：85%+
            
            Month 7-8  ► 预测Agent轻量模型训练
            Month 9-10 ► 跨系统集成（项目管理/薪酬）
            Month 11-12► Agent市场 + 自定义Agent

### 6.3 MVP 最小可行产品（Month 1-2）

```
【MVP自动化程度分层】

Tier 1: 全自动（无需人工确认）
├─ 每日学习套餐推送（Content Agent自动组装）
├─ Coach Agent苏格拉底对话（全自动）
├─ 行为埋点自动记录（全自动）
├─ 能力图谱自动更新（基于规则自动计算）
└─ 预测Agent规则预警（全自动，仅通知导师）

Tier 2: 半自动（人工抽检，非全量确认）
├─ 诊断Agent作品分析 → 系统自动生成结果，导师抽检20%
├─ 学习路径推荐 → 系统自动推送，导师仅在"异常"时介入
└─ 任务完成状态 → 系统自动标记，导师抽检10%

Tier 3: 人工确认（必须人工审核）
├─ 能力图谱正式发布（影响晋升）
├─ 学员风险升级（🟡→🔴）
└─ 导师对Agent结果的异议申诉

【修订后的MVP闭环】

学员入职 ──► 诊断Agent分析作品 ──► 自动生成结果
                                    │
                                    ▼（导师抽检20%，发现问题可修正）
              内容Agent推荐路径 ──► 自动推送
                                    │
                                    ▼（导师仅在"异常"时介入）
              CoachAgent推送任务 ──► 学员在飞书完成
                                    │
                                    ▼（自动标记完成）
                              能力图谱自动更新
                                    │
                                    ▼（月度导师Review时人工确认）
                              发布正式能力评估

MVP 必须包含（Month 1-2）：
  ✅ Orchestrator + Memory + Tool Registry
  ✅ 诊断Agent（作品分析 + 微场景测试）
  ✅ 内容Agent（基础路径组装）
  ✅ CoachAgent（基础苏格拉底对话）
  ✅ 预测Agent（规则层预警：连续3天未登录/停留时间>200%/入职7天完成<60%）
  ✅ External Knowledge轻量版（多维表格关键词索引）
  ✅ 飞书消息推送 + 飞书待办
  ✅ 能力图谱可视化（人工审核后发布）

Phase 2 必须包含（Month 3-5，新增）：
  ✅ 360度评估系统（飞书审批流 + 自动汇总 + 季度成长报告）
  ✅ 等级认证体系（晋升审批流 + 能力图谱校验 + 电子徽章）
  ✅ 导师激励体系（积分自动化 + 排行榜 + 兑换审批）
  
MVP 不包含：
  ❌ AI Assessment Generator（人工出题）
  ❌ 实战Agent（飞书文档替代）
  ❌ 智能派活推荐（人工查询能力图谱替代）
  ❌ 360度评估同事评价（MVP匿名机制未就绪）
  ❌ 导师激励与薪酬系统对接（Out of Scope）
```

## 七、风险应对

### 7.1 技术风险

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|---------|
| **Agent 幻觉导致误导** | 中 | 高 | ① RAG强制检索，回答必须基于知识库 ② Human-in-the-Loop关键决策人工审核 ③ 置信度阈值：低置信度标记「待核实」 ④ 建立Agent决策日志审计机制 |
| **Agent 协作冲突** | 中 | 中 | Orchestrator统一仲裁 + Memory共享 + 冲突解决规则库。MVP阶段限制Agent并发数量，逐步扩展 |
| **LLM 供应商故障** | 中 | 高 | 设计「降级模式」：LLM不可用时规则引擎接管关键功能。多供应商备选（百炼/豆包/Claude） |
| **飞书API限流导致系统不可用** | **中** | **高** | **异步队列削峰 + 本地缓存 + 批量写入 + 飞书API降级预案** |
| **LLM推理延迟导致用户体验差** | **中** | **中** | **异步决策模式 + 默认使用快速模型（qwen-turbo）+ 规则引擎优先** |
| **系统与飞书整合困难** | 低 | 中 | 优先使用飞书原生能力，Agent层完全基于飞书生态构建 |

### 7.2 业务风险

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|---------|
| **资深员工不愿意分享** | 中 | 高 | 设计激励机制（积分/奖金/荣誉），让分享=收益。Agent辅助降低分享成本：口述→Agent自动整理为结构化案例 |
| **Tim没有时间配合拆解案例** | 高 | 高 | 先让资深剪辑师/编导拆解，Tim的爆款作为「特别篇」。Agent辅助：上传素材和成片，Agent自动提取时间轴、生成初版拆解框架，专家只需审核修正 |
| **新人觉得系统枯燥不用** | 低 | 中 | v2.0从根本上解决：Agent主动推送，学员无需「打开系统找课」。对话式交互天然比「点击课程列表」更有趣 |
| **学员对 Agent 不信任** | 中 | 中 | 透明度设计：每次推荐说明理由。渐进式信任：MVP阶段Agent只做「推荐」不做「决策」 |
| **数据隐私与合规** | 低 | 高 | 最小权限原则 + 数据脱敏 + 本地化处理。学员作品分析数据仅限培训系统，不与绩效直接挂钩 |

### 7.3 项目风险

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|---------|
| **内容库建设滞后** | 中 | 高 | 内容建设与系统开发并行推进；MVP阶段先填充「必修」课程；案例拆解由Agent辅助生成初版 |
| **Prompt调优时间超预期** | **高** | **中** | **预留4周专项时间 + 配备专门Prompt工程师1人 + 三阶段Prompt评估机制（离线评估→内部验证→周级全量切换）** |
| **AI Assessment概念题准确率不达标** | **低** | **中** | **按题型分层设目标 + 人工兜底（实操/追问题型改为人工出题）** |
| **【新增】新增模块（360度/认证/激励）设计与实施脱节** | **中** | **高** | **v4.1已将三个模块明确排入路线图和MVP边界；实施中每2周Review一次进度** |
| **【新增】预测Agent规则层命中率低于预期** | **中** | **中** | **设定保守基线45-55%，明确Phase 2模型目标≥60%；规则层仅预警不决策，降低误报影响** |
| **【新增】轻量RAG关键词索引命中率不足** | **中** | **中** | **Phase 0 PoC验证索引命中率，目标>70%；不达标则扩充关键词库或提前引入语义检索** |
| **用户接受度低** | 低 | 高 | 试点先行：选择2-3名积极的新人试用，收集反馈快速迭代；导师端先展示「时间节省」价值 |
| **需求蔓延** | 中 | 中 | 严格执行Phase划分，每Phase结束后Review；新增需求放入Backlog，不干扰当前Sprint |
| **实战Agent Phase 2技术风险** | **中** | **高** | **MVP用飞书消息替代降低风险；Phase 2提前3个月启动技术预研** |
