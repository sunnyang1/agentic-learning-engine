设计评审方法论

建立四维度评审框架，每轮评审聚焦架构可行性、性能达标性、Prompt 预研必要性、AI 准确率现实性。形成「评审→修正→新版本→再评审」的标准化闭环。

评审框架

四维度评审法

🏗
架构可行性
技术选型是否匹配飞书生态？新增模块与现有架构的耦合度？数据模型是否能支撑业务场景？
⚡
性能达标性
延迟指标是否可验证？新增调用链路的延迟预算是否拆分？飞书 Webhook 固有延迟是否考虑？
📝
Prompt 预研必要性
新增 AI 生成场景是否已纳入 Prompt 管理？三阶段评估机制（离线→验证→全量）是否覆盖？
🎯
AI 准确率现实性
预测 Agent/自动评估的准确率预期是否有基准数据？规则层命中率是否设定保守基线？

典型问题案例

从问题发现到闭环修正

🔴
P0 案例：性能指标下调
P0
问题：方案将需求中的规则引擎延迟 < 100ms 调整为 < 200ms，未解释原因。

根因：飞书 Webhook 固有延迟 500ms-2s，内部执行 < 200ms 已接近极限。

修正：拆分为两层指标——内部执行延迟 + 端到端延迟，并解释与需求差距的根因。建议 Phase 2 评估将规则引擎部署在飞书应用引擎内的优化方案。
🔴
P0 案例：新增模块未纳入路线图
P0
问题：v4.0 新增 360度评估、等级认证、导师激励 3 个模块（164行设计），但未在路线图和 MVP 中排期。

根因：需求跟踪矩阵评审要求补充设计，但未同步更新实施计划。

修正：v4.1 明确排期（360度评估 Month 3-4、认证/激励 Month 4-5），总工期从 28周调整至 33周。
🟡
P1 案例：预测 Agent 命中率基准缺失
P1
问题：规则层（硬阈值）的典型命中率 40-60%，但需求目标 >60%，MVP 可能不达标。

修正：设定保守基线 45-55%，明确 Phase 2 轻量模型（XGBoost）目标 ≥60%，需 4 个月+100+样本训练数据。
🟡
P1 案例：轻量 RAG 延迟未拆分
P1
问题：Phase 1 用飞书云文档搜索+关键词索引替代 RAG，但未设定延迟预算和命中率目标。

修正：增加延迟预算表（关键词命中路径内部<3.2s/端到端<5s，目标命中率>70%；降级路径内部<5s/端到端<7s）。
