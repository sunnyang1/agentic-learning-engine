## 一、文档说明

### 1.1 目的

本文档为 MediaStorm 新人培训与能力图谱系统（Agentic Learning v2.0）制定完整的运营方案，涵盖从上线策略到日常运营、从内容生产到 Agent 调优、从用户活跃到效果评估的全链路运营动作。目标是确保系统不仅「上线」，更能「用好」「用出效果」，真正支撑 MediaStorm 的人才战略目标。

### 1.2 运营核心目标

| 目标维度 | 具体目标 | 衡量指标 | 达成时间 |
|----------|----------|----------|----------|
| ** adoption（采纳）** | 100% 新人使用系统完成入职培训 | 新人系统使用率 | Month 1 |
| **engagement（活跃）** | 学员日均学习时长 > 30 分钟 | 平均学习时长 | Month 2 |
| **effectiveness（效果）** | 新人独立产出周期从 2 个月缩短到 2 周 | 时间到胜任 | Month 3 |
| **retention（留存）** | 导师参与率 > 60%，学员满意度 NPS > 50 | 导师参与率、NPS | Month 3 |
| **evolution（进化）** | Agent 自主决策率从 20% 提升到 70% | 自主决策率 | Month 6 |

### 1.3 运营团队角色

| 角色 | 职责 | 人数（建议） |
|------|------|-------------|
| **运营负责人** | 整体运营策略制定、跨部门协调、效果汇报 | 1 人 |
| **内容运营** | 课程/案例/资源的生产、审核、更新 | 1-2 人 |
| **Agent 运营（新型角色）** | Agent 决策日志审查、策略调优、Human-in-the-Loop 管理 | 1 人 |
| **用户运营** | 学员/导师 onboarding、活跃度维护、反馈收集 | 1 人 |
| **数据分析师** | 埋点数据监控、报表制作、A/B 测试分析 | 0.5 人（可兼职） |
| **技术支持** | 系统稳定性维护、飞书生态对接、故障处理 | 1 人 |

## 二、Launch Strategy（上线策略）

### 2.1 上线阶段划分

```
上线路线图
│
├── Phase 0：内部试运营（Week -2 ~ Week 0）
│   └── 对象：产品/技术团队 + 3 名志愿导师 + 5 名模拟学员
│
├── Phase 1：Soft Launch 软启动（Week 1-4）
│   └── 对象：剪辑师团队（30 人），其中新人 5 人
│
├── Phase 2：扩展上线（Week 5-8）
│   └── 对象：编导团队（20 人）+ 管理层看板
│
├── Phase 3：全面上线（Week 9-12）
│   └── 对象：全员（100 人），含摄影师/制片等二期岗位
│
└── Phase 4：持续运营（Month 4+）
    └── 全功能稳定运行，进入迭代优化周期
```

### 2.2 Pilot Group（试点组）选择

#### 2.2.1 试点组筛选标准

| 维度 | 标准 | 理由 |
|------|------|------|
| **岗位** | 优先剪辑师（人数最多、标准化最高） | PRD 定义 MVP 首选岗位 |
| **意愿度** | 导师和学员均自愿参与 | 降低抵触情绪，获得真实反馈 |
| **代表性** | 覆盖 L1-L3 等级 | 验证系统对不同层级的效果 |
| **技术接受度** | 愿意使用飞书及新工具 | 确保基础使用无障碍 |
| **业务压力** | 当前项目负荷适中 | 避免因太忙而无法配合 |

#### 2.2.2 首批试点名单建议（15 人）

| 角色 | 人数 | 选择标准 |
|------|------|----------|
| 新人剪辑师（< 3 个月） | 3 人 | 直接验证新人培训效果 |
| 初级剪辑师（3-12 个月） | 2 人 | 验证成长路径功能 |
| 资深剪辑师（导师候选人） | 3 人 | 验证导师辅助功能 |
| 剪辑指导（高级导师） | 1 人 | 验证管理看板 |
| 制片人 | 2 人 | 验证智能派活功能 |
| HR/培训负责人 | 1 人 | 验证培训管理后台 |
| 管理层（Tim 等） | 1 人 | 验证团队洞察看板 |
| 产品/技术内部人员 | 2 人 | 实时收集 bug 和体验问题 |

### 2.3 Rollout 计划（按团队/角色）

#### 2.3.1 第一阶段：剪辑师团队（Week 1-4）

| 周次 | 动作 | 目标 | 成功标准 |
|------|------|------|----------|
| **Week 1** | 系统部署 + 数据初始化 | 导入现有课程、案例、学员信息 | 系统可正常访问，无阻断性 bug |
| **Week 2** | 试点组 onboarding + 诊断 Agent 试运行 | 5 名新人完成诊断，3 名导师熟悉预审看板 | 诊断报告被导师认可率 > 70% |
| **Week 3** | 内容 Agent + Coach Agent 上线 | 每日推送学习套餐，Coach 处理基础提问 | 学员日均使用 > 3 天 |
| **Week 4** | 预测 Agent 上线 + 首次复盘 | 预警机制运行，收集第一轮反馈 | 预警命中率 > 50%（初期） |

#### 2.3.2 第二阶段：编导团队 + 管理层（Week 5-8）

| 周次 | 动作 | 目标 | 成功标准 |
|------|------|------|----------|
| **Week 5** | 编导岗位能力模型配置 | 定义编导能力图谱和等级标准 | 能力模型被编导团队认可 |
| **Week 6** | 编导团队 onboarding | 10 名编导接入系统 | 使用率 > 80% |
| **Week 7** | 管理层看板上线 | Tim 等可查看团队能力热力图 | 管理层每周查看 > 1 次 |
| **Week 8** | 跨岗位协作验证 | 验证制片人智能派活 + 导师跨岗位匹配 | 派活推荐采纳率 > 60% |

#### 2.3.3 第三阶段：全员覆盖（Week 9-12）

| 周次 | 动作 | 目标 | 成功标准 |
|------|------|------|----------|
| **Week 9-10** | 摄影师/制片等二期岗位接入 | 扩展能力图谱到更多岗位 | 新岗位学员可正常诊断 |
| **Week 11** | AI Assessment Generator 全量上线 | 核心课程均有 AI 生成测试题 | 题目审阅通过 > 70% |
| **Week 12** | 全量数据复盘 + Phase 1 验收 | 输出 Month 3 运营报告 | 核心指标达 Phase 1 目标 |

## 三、Content Operations（内容运营）

### 3.1 内容生产体系

```
内容生产体系
│
├── 标准化课程（Standard Courses）
│   ├── 生产主体：资深员工 + Agent 辅助
│   ├── 更新频率：每月 2-4 门新课
│   └── 质量标准：学习目标明确、时长 3-25 分钟、有配套测试题
│
├── 案例拆解库（Case Studies）
│   ├── 生产主体：项目团队 + Agent 辅助生成初稿
│   ├── 更新频率：每月 2 个新案例
│   └── 质量标准：6 维度拆解、有时间轴标注、可下载学习清单
│
├── 资源库（Assets）
│   ├── 生产主体：品牌/法务/技术团队
│   ├── 更新频率：按需更新，版权到期前 30 天预警
│   └── 质量标准：版权清晰、格式统一、带使用说明
│
└── Skill Documents（程序性记忆）
    ├── 生产主体：Agent 自动生成 + 运营审核
    ├── 更新频率：每次成功干预后自动生成/更新
    └── 质量标准：触发条件清晰、策略可复现、有效果数据
```

### 3.2 内容生产 SOP

#### 3.2.1 课程生产流程

```
Step 1：需求提出（每月第 1 周）
  └─ 来源：学员 FAQ 高频问题、导师反馈短板、新项目需求、行业趋势
  └─ 工具：《内容需求收集表》（飞书多维表格）

Step 2：Owner 认领（每月第 1 周）
  └─ 每个课程指定 1 名 Owner（资深员工）
  └─ Owner 负责：内容准确性、案例真实性、符合公司标准

Step 3：内容制作（每月第 2-3 周）
  └─ 形式：视频（推荐 3-15 分钟）、图文手册、互动测试
  └─ 辅助：Agent 可帮助生成课程大纲、提取关键点

Step 4：AI 测试题生成（每月第 3 周）
  └─ AI Assessment Generator 自动生成 6-12 题
  └─ 进入 Pending Pool

Step 5：Owner 审阅（每月第 3-4 周）
  └─ Owner 审阅课程 + 测试题
  └─ 审阅 Checklist：知识点对齐、难度适合、无歧义

Step 6：发布上线（每月第 4 周）
  └─ 课程发布 → 测试题 Released → 内容 Agent 可推荐
  └─ 飞书全员通知：「本月新课上线」
```

#### 3.2.2 案例拆解生产流程

```
Step 1：爆款筛选（每月初）
  └─ 标准：播放量 > 100 万 或 内部认定「值得学习」
  └─ 来源：B站数据 + 内部项目库 + 管理层推荐

Step 2：多角色访谈（每月第 1-2 周）
  └─ 访谈：导演、剪辑师、制片、数据分析师
  └─ Agent 辅助：自动提取项目数据、生成访谈提纲

Step 3：初稿撰写（每月第 2 周）
  └─ 使用统一模板（6 维度）
  └─ Agent 辅助：上传成片和原始素材，自动提取时间轴、生成初版框架

Step 4：专家审核（每月第 3 周）
  └─ 由该案例的导演/剪辑师审核
  └─ 重点：技巧描述准确、关键帧截图正确、数据真实

Step 5：发布与推广（每月第 4 周）
  └─ 发布到案例库
  └─ 内容 Agent 自动识别相关学员并推送
  └─ 飞书群讨论：「本月案例拆解《XXX》，欢迎交流」
```

### 3.3 内容质量标准

| 内容类型 | 质量维度 | 标准 | 检查方式 |
|----------|----------|------|----------|
| **视频课程** | 时长 | 3-25 分钟，单一知识点 | 自动检测 |
| | 画质 | 1080P+，字幕清晰 | 人工抽查 |
| | 学习目标 | 每门课有 1-3 个明确目标 | Owner 审核 |
| | 实用性 | 与真实工作场景相关 | 学员评分 |
| **案例拆解** | 完整性 | 6 维度齐全 | 模板检查 |
| | 可复现性 | 技巧可应用到其他项目 | 导师评审 |
| | 时效性 | 案例发布时间在 2 年内 | 自动检测 |
| **测试题** | 考点对齐 | 与课程内容一致 | AI 自检 + Owner 审核 |
| | 难度合理 | 易:中:难 ≈ 3:4:3 | AI 自检 |
| | 无歧义 | 有唯一/明确最佳答案 | Owner 审核 |
| | 开放性题 | 创意类题目允许开放答案 | Owner 标注 |

## 四、Agent Operations（Agent 运营）

### 4.1 Agent 运营定位

Agent 运营是 Agentic Learning 系统的独特运营岗位，负责：
- 监控 Agent 的「健康状况」（决策质量、响应速度、错误率）
- 管理 Human-in-the-Loop 流程（人工审核、反馈、修正）
- 持续优化 Agent 策略（基于数据调参、更新 Prompt、调整规则）
- 治理 Skill Document（审核、版本管理、质量把控）

### 4.2 Agent 性能监控日报

| 监控项 | 正常范围 | 黄色预警 | 红色告警 | 监控频率 |
|--------|----------|----------|----------|----------|
| Orchestrator 决策延迟 | < 3 秒 | 3-5 秒 | > 5 秒 | 实时 |
| Agent 消息响应时间 | < 5 秒 | 5-10 秒 | > 10 秒 | 实时 |
| Agent 自主决策率 | 符合当前 Phase 目标 | 偏离目标 ±10% | 偏离目标 ±20% | 每日 |
| LLM API 调用成功率 | > 99% | 95-99% | < 95% | 实时 |
| 预测 Agent 预警命中率 | > 60% | 50-60% | < 50% | 每周 |
| Coach Agent 对话满意度 | > 4.0 | 3.5-4.0 | < 3.5 | 每周 |
| 内容 Agent 路径完成率 | > 60% | 50-60% | < 50% | 每周 |
| Memory 读写失败率 | < 0.1% | 0.1-1% | > 1% | 实时 |
| Tool Registry 调用失败率 | < 0.5% | 0.5-2% | > 2% | 实时 |
| 系统整体错误率 | < 0.1% | 0.1-1% | > 1% | 实时 |

### 4.3 Human-in-the-Loop 运营流程

```
Human-in-the-Loop 运营流程
│
├── 每日审查（15 分钟/天）
│   └── 审查项：Agent 决策日志中标记为「需人工审核」的记录
│   └── 重点：高风险预警、能力等级重大调整、导师匹配变更
│
├── 每周审查（1 小时/周）
│   └── 审查项：Agent 冲突仲裁记录、被学员投诉的 Agent 决策
│   └── 重点：Agent 是否给出矛盾建议、是否有明显错误
│
├── 每月深度审查（2 小时/月）
│   └── 审查项：Skill Document 质量、Agent 自学习效果
│   └── 重点：Skill Doc 是否有错误策略、Agent 是否从反馈中正确学习
│
└── 季度审计（半天/季度）
    └── 审查项：Agent 决策的公平性、隐私合规性
    └── 重点：是否存在偏见（如对某类学员系统性推荐更难内容）
```

### 4.4 Agent 策略调优机制

#### 4.4.1 调优触发条件

| 触发条件 | 动作 | 负责人 |
|----------|------|--------|
| 预警命中率连续 2 周 < 60% | 调整预测阈值、优化特征工程 | Agent 运营 + 数据分析师 |
| Coach Agent 满意度连续 2 周 < 3.8 | 优化苏格拉底 Prompt、调整对话深度评估标准 | Agent 运营 |
| 内容 Agent 路径完成率连续 2 周 < 55% | 调整路径组装算法、增加内容库匹配度权重 | Agent 运营 + 内容运营 |
| 诊断 Agent 与导师评估一致率 < 65% | 优化作品分析 Prompt、增加评估维度 | Agent 运营 + 技术 |
| 学员反馈「推荐不精准」> 10 条/周 | 分析反馈模式、调整个性化算法 | Agent 运营 |
| 系统错误率突增 | 紧急排查、回滚到上一稳定版本 | 技术支持 |

#### 4.4.2 Prompt 版本管理

```
Prompt 版本管理规范
│
├── 命名规则：{AgentName}_{Purpose}_v{Major}.{Minor}
│   例：CoachAgent_Socratic_v2.1
│
├── 版本控制：
│   ├── Major 升级：策略大幅调整（如从直接给答案改为苏格拉底式）
│   └── Minor 升级：措辞优化、边界 case 修复
│
├── 变更流程：
│   1. Agent 运营提出变更需求
│   2. 在 Staging 环境测试（100 条模拟对话）
│   3. A/B 测试（5% 流量）运行 3 天
│   4. 指标达标后全量发布
│   5. 回滚方案准备（保留上一版本 Prompt）
│
└── 文档要求：
    ├── 每个版本记录：变更原因、变更内容、测试数据、发布日期
    └── 存储位置：飞书云文档「Agent Prompt 仓库」
```

### 4.5 Skill Document 治理

#### 4.5.1 Skill Doc 生命周期

```
生成 → 审核 → 发布 → 复用 → 评估 → 更新/废弃
```

| 阶段 | 动作 | 负责人 | SLA |
|------|------|--------|-----|
| **生成** | Agent 成功干预后自动合成 | Agent | 干预成功后 1 小时内 |
| **审核** | Agent 运营审核 Skill Doc 的合理性和安全性 | Agent 运营 | 生成后 24 小时内 |
| **发布** | 审核通过后标记为 active | Agent 运营 | 审核通过后即时 |
| **复用** | Agent 在类似场景自动读取 | Agent | 实时 |
| **评估** | 每月统计 Skill Doc 的复用率和成功率 | 数据分析师 | 每月 |
| **更新** | 复用后对比效果，自动/手动更新 | Agent | 每次复用后 |
| **废弃** | 成功率 < 50% 或 6 个月无复用，标记 deprecated | Agent 运营 | 每月清理 |

#### 4.5.2 Skill Doc 质量标准

| 检查项 | 标准 | 未达标处理 |
|--------|------|------------|
| 触发条件清晰度 | 条件可被机器解析执行 | 退回 Agent 重新生成 |
| 策略安全性 | 不包含可能伤害学员的操作 | 立即废弃，人工介入 |
| 效果数据完整性 | 包含成功率、平均解决时间 | 补充数据后发布 |
| 版本历史 | 每次更新有明确记录 | 补充版本说明 |
| 去标识化 | 不包含学员真实姓名等敏感信息 | 脱敏后发布 |

| 项目完成 | 「这期视频的完播率比上一条高 12%，哪个决策起了作用？」 | 项目数据回传时 |

#### 5.2.3 Recognition（认可）

| 认可类型 | 方式 | 频率 |
|----------|------|------|

### 5.3 低 adoption 与阻力应对

#### 5.3.1 阻力类型与应对

| 阻力类型 | 表现 | 根因分析 | 应对策略 |
|----------|------|----------|----------|
| **「不需要」** | 资深员工认为系统对TA没用 | 未感受到价值 | 强调导师积分、荣誉体系；展示预审看板节省的时间 |
| **「不信任 AI」** | 质疑 Agent 推荐的准确性 | 对 AI 不了解或有负面经历 | 透明度设计：每次推荐说明理由；渐进式信任 |
| **「太忙」** | 以工作繁忙为由不使用 | 时间成本高 | 强调系统节省时间的价值；将学习嵌入工作流 |
| **「太麻烦」** | 觉得系统操作复杂 | onboarding 不充分 | 优化 onboarding；提供 1 对 1 引导 |
| **「隐私担忧」** | 担心作品/能力数据被滥用 | 对数据用途不清楚 | 明确告知数据使用范围；提供查看/删除入口 |
| **「习惯旧方式」** | 喜欢原来的带教/学习方式 | 改变习惯需要成本 | 不强制切换；并行运行 1 个月；对比展示效果 |

#### 5.3.2 沉默用户激活

| 沉默类型 | 定义 | 激活策略 |
|----------|------|----------|
| **新用户沉默** | 注册后 3 天内未开始新手村 | 导师 1 对 1 邀请；HR 电话跟进；简化首日任务 |
| **学习中断** | 连续 3 天无学习行为 | Agent 自动干预 → 导师介入 → HR 关怀 |
| **内容不消费** | 登录但不点击任何内容 | 内容 Agent 调整推荐策略；询问偏好 |
| **测评不参与** | 完成课程但不做测评 | 降低测评门槛；强调测评价值（能力图谱更新） |
| **导师不活跃** | 分配导师后无辅导记录 | 提醒导师；提供辅导模板；纳入导师考核 |

## 六、Feedback Collection（反馈收集）

### 6.1 反馈收集体系

```
反馈收集体系
│
├── 定量反馈（Quantitative）
│   ├── 系统内埋点（自动采集）
│   ├── 满意度评分（CSAT）
│   ├── 净推荐值（NPS）
│   └── 结构化问卷
│
├── 定性反馈（Qualitative）
```

### 6.2 定量反馈

#### 6.2.1 系统内自动采集

| 反馈信号 | 采集方式 | 说明 |
|----------|----------|------|
| **内容完成率** | 埋点自动采集 | 完成率 < 50% 视为对内容不感兴趣 |
| **消息响应速度** | 埋点自动采集 | 响应变慢可能表示疲劳或不满 |
| **路径调整接受率** | 埋点自动采集 | 拒绝率 > 30% 说明推荐不精准 |
| **通知点击率** | 埋点自动采集 | 点击率 < 20% 说明推送策略需优化 |
| **页面停留时间** | 埋点自动采集 | 异常短/长都有反馈价值 |

#### 6.2.2 满意度评分（CSAT）

| 触发场景 | 问题 | 评分方式 |
|----------|------|----------|
| 完成 Coach Agent 对话后 | 「这次对话对你有帮助吗？」 | 1-5 星 |
| 完成每日学习套餐后 | 「今天的学习内容合适吗？」 | 1-5 星 |
| 导师辅导结束后 | 「你对这次辅导满意吗？」 | 1-5 星 |
| 收到预警干预后 | 「这次提醒及时吗？」 | 1-5 星 |
| 完成测评后 | 「题目难度合适吗？」 | 太简单/刚好/太难 |

#### 6.2.3 净推荐值（NPS）

| 调查对象 | 频率 | 问题 |
|----------|------|------|
| 学员 | 每月 | 「你有多大可能向新同事推荐这个学习系统？（0-10 分）」 |
| 导师 | 每季度 | 「你有多大可能向其他资深同事推荐成为导师？（0-10 分）」 |
| 管理层 | 每季度 | 「这个系统对团队管理有多大帮助？（0-10 分）」 |

**NPS 计算**：
- 推荐者（9-10 分）：% Promoters
- 被动者（7-8 分）：% Passives
- 贬损者（0-6 分）：% Detractors
- **NPS = % Promoters - % Detractors**

#### 6.2.4 结构化问卷

| 问卷名称 | 发放时机 | 题目数 | 覆盖范围 |
|----------|----------|--------|----------|
| **新手村毕业问卷** | Day 7 完成测评后 | 8 题 | 系统易用性、内容质量、Agent 帮助度 |
| **月度学习体验问卷** | 每月最后一周 | 10 题 | 学习节奏、内容匹配度、导师支持、系统稳定性 |
| **导师体验问卷** | 每季度 | 6 题 | Agent 辅助效果、时间节省、带教成就感 |
| **管理层满意度问卷** | 每季度 | 5 题 | 数据洞察价值、团队能力可见性、决策支持 |

## 七、Iteration Cycle（迭代节奏）

### 7.1 迭代节奏总览

```
迭代节奏
│
├── 每日（Daily）
│   ├── 数据监控：Agent 性能、系统稳定性、用户活跃
│   └── 问题响应：处理用户反馈的紧急问题
│
├── 每周（Weekly）
│   ├── 数据回顾：核心指标周环比
│   ├── Agent 调优：Prompt 微调、阈值调整
│   └── 内容更新：新课程/案例上线
│
├── 每月（Monthly）
│   ├── 深度分析：月度运营报告
│   ├── 用户访谈：收集定性反馈
│   ├── A/B 测试：启动/结束实验
│   └── 版本发布：功能更新上线
│
└── 每季度（Quarterly）
    ├── 战略复盘：季度业务复盘
    ├── 能力图谱校准：与导师对齐等级标准
    ├── Agent 成熟度升级：从 Phase N 到 Phase N+1
    └── 技术债务清理：性能优化、架构升级
```

### 7.2 周度运营节奏（Weekly Cadence）

#### 周一：计划日

| 时间 | 动作 | 产出 |
|------|------|------|
| 09:00-09:30 | 查看上周核心指标仪表盘 | 指标异常清单 |
| 09:30-10:00 | 周会：技术 + 运营 + 内容 | 本周优先级对齐 |
| 10:00-10:30 | 内容运营：确认本周上线内容 | 内容发布计划 |
| 10:30-11:00 | Agent 运营：审查周末 Agent 决策日志 | 需关注的问题清单 |

#### 周三：执行日

| 时间 | 动作 | 产出 |
|------|------|------|
| 14:00-15:00 | 用户运营：处理用户反馈 | 反馈处理记录 |
| 15:00-16:00 | Agent 运营：Prompt 微调/阈值调整 | 配置更新记录 |
| 16:00-17:00 | 数据分析师：A/B 测试中期数据 | 实验进展报告 |

#### 周五：复盘日

| 时间 | 动作 | 产出 |
|------|------|------|

### 7.4 优先级排序框架

```
优先级排序 — RICE 框架（适配版）

Reach（影响范围）：该改进会影响多少用户？
  ├── 全员（100 人）= 5 分
  ├── 某岗位（30 人）= 3 分
  └── 少数人（< 10 人）= 1 分

Impact（影响深度）：对核心指标的提升潜力？
  ├── 大幅提升（> 20%）= 5 分
  ├── 中等提升（10-20%）= 3 分
  └── 小幅提升（< 10%）= 1 分

Confidence（信心度）：对效果的确信程度？
  ├── 高（有数据支撑/竞品验证）= 100%
  ├── 中（有假设但需验证）= 70%
  └── 低（纯猜测）= 40%

Effort（投入成本）：需要多少人力/时间？
  ├── 低（< 1 人周）= 1 分
  ├── 中（1-2 人周）= 2 分
  └── 高（> 2 人周）= 3 分

RICE Score = (Reach × Impact × Confidence) / Effort
```

```
版本发布流程

│   ├── 16:00  全量发布
│   └── 18:00  发布日报
│
└── 发布后 1 天
    ├── 核心指标对比（发布前 vs 发布后）

    └── 如有问题，启动回滚
```

### 8.2 关键里程碑与 Go/No-Go 标准

#### Phase 1（Month 1-2）：基础设施 — Go/No-Go

| 检查项 | Go 标准 | No-Go 风险 |

### 9.2 应急预案

#### 场景 1：系统大面积故障

```
T+0 分钟：自动告警触发，技术支持响应
T+5 分钟：确认故障范围，启动降级模式
T+10 分钟：飞书群通知全员：「系统维护中，预计 X 分钟恢复」
T+30 分钟：如未恢复，启用备用方案（导师人工安排学习）
T+60 分钟：提供故障进展更新
恢复后：发送恢复通知 + 故障复盘报告
```

#### 场景 2：Agent 给出严重错误建议

```
发现后 5 分钟内：立即屏蔽该 Agent 的该类建议
发现后 15 分钟内：通知可能受影响的学员/导师，澄清正确信息
发现后 1 小时内：排查根因（Prompt 问题？数据问题？LLM 幻觉？）
发现后 4 小时内：修复并部署
24 小时内：发送事件报告，记录到 Agent 决策日志
```

#### 场景 3：学员数据泄露

```
发现后 立即：隔离泄露源，停止相关服务
发现后 30 分钟：通知管理层和法务
发现后 1 小时：评估影响范围，通知受影响用户
24 小时内：提交初步事件报告
72 小时内：完成修复，提交详细复盘报告
```
