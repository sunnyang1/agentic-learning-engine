## 三、Sprint 详细设计

### Sprint 1：飞书基础能力上手（Week 1-2）

**Sprint目标**：团队熟悉飞书生态，建立开发规范，产出可复用Workflow模板

**用户故事**：
1. 作为技术团队，我想要熟悉飞书应用引擎Workflow，以便后续快速搭建Agent调度逻辑
2. 作为技术团队，我想要掌握飞书多维表格API读写，以便实现Memory层数据持久化
3. 作为技术团队，我想要配置飞书机器人消息推送，以便实现Agent主动触达学员

**任务分解**：
- [ ] Day 1-2：飞书应用引擎Workflow基础（条件分支/循环/延时节点）
- [ ] Day 3-4：飞书多维表格API读写 + 自动化流程配置
- [ ] Day 5：飞书机器人消息推送（文本/卡片/按钮）
- [ ] Day 6-7：飞书应用引擎调用外部API（百炼LLM）
- [ ] Day 8-9：飞书智能伙伴配置 + 知识库接入
- [ ] Day 10：团队PoC Demo（简单Agent对话闭环）

**验收标准**：
- ✅ 飞书机器人可向指定用户发送文本消息和卡片消息
- ✅ 可通过API读写多维表格记录（CRUD）
- ✅ 可调用百炼qwen-turbo并返回结果
- ✅ 产出《飞书开发规范文档》v1.0
- ✅ 产出3个可复用Workflow模板（定时触发/事件触发/审批触发）

**风险**：
- ⚠️ 飞书免费版功能限制 → 对策：提前确认Base版能力边界，高级功能在方案中描述

### Sprint 2：PoC验证（Week 3-4）

**Sprint目标**：验证关键技术方案可行性，消除架构不确定性

**用户故事**：
1. 作为技术负责人，我想要验证轻量RAG的关键词索引命中率，以便确认Phase 1方案可行
2. 作为技术负责人，我想要验证百炼LLM对影视剪辑领域的理解能力，以便确认CoachAgent可行
3. 作为技术负责人，我想要验证飞书Webhook+异步队列的延迟表现，以便确认性能指标可达

**任务分解**：
- [ ] Day 1-2：轻量RAG PoC（构建50条关键词索引 → 测试20个查询 → 统计命中率）
- [ ] Day 3-4：LLM领域测试（向qwen-turbo提交10个影视剪辑专业问题 → 评估回答质量）
- [ ] Day 5-6：延迟测试（模拟50并发事件 → 测量Webhook触发→Workflow执行→消息推送全链路延迟）
- [ ] Day 7-8：飞书审批流PoC（创建测试审批模板 → 模拟季度评估审批流程）
- [ ] Day 9-10：PoC总结 + 技术方案微调

**验收标准**：
- ✅ 轻量RAG关键词索引命中率 ≥60%（目标>70%，若<60%需调整方案）
- ✅ LLM对8/10个影视剪辑问题给出专业回答
- ✅ 规则引擎端到端延迟 <3s（80%场景）
- ✅ 飞书审批流可在3步内完成（申请→审批→通知）
- ✅ 产出《PoC验证报告》+ 技术方案微调建议

**风险**：
- ⚠️ 轻量RAG命中率不达标 → 对策：扩充关键词库或提前引入语义检索
- ⚠️ LLM对影视领域理解不足 → 对策：准备领域特定的System Prompt和Few-shot示例

**任务分解**：

- [ ] Content Tools封装：search_course / search_case / get_resource / assemble_learning_package

- [ ] Assessment Tools封装：generate_questions / submit_quiz / get_skill_gap / evaluate_work（Mock）

- ✅ Tool Registry表在多维表格中可配置，支持增删改查

### Sprint 6：Memory两层架构（Week 11-12）

**Sprint目标**：搭建Hot Memory + Cold Storage数据层

**用户故事**：
1. 作为后端开发，我想要创建User/LearningRecord/AgentDecisionLog等多维表格，以便存储结构化数据
2. 作为后端开发，我想要实现Hot Memory的Nudge更新机制，以便保持LLM前缀缓存稳定
3. 作为后端开发，我想要配置Cold Storage分层（类型A/B/C），以便按需存储不同生命周期数据

**任务分解**：
- [ ] 创建9张多维表格：User / LearningRecord / AgentDecisionLog / QuarterlyReview / LevelCertification / MentorPoints / MentorPointsHistory / Course / CaseStudy
- [ ] Hot Memory：USER.md（~500tokens）+ MEMORY.md（~800tokens）的JSON字段设计 + Nudge触发逻辑
- [ ] Cold Storage类型A：结构化数据（AgentDecisionLog/学习记录摘要）→ 多维表格
- [ ] Cold Storage类型B：非结构化数据（对话记录/作品分析）→ 飞书云文档
- [ ] Cold Storage类型C：归档数据（超6个月）→ 阿里云OSS
- [ ] 数据归档自动化规则：超6个月数据自动迁移到OSS

**验收标准**：
- ✅ 9张多维表格字段完整，测试数据可正常读写
- ✅ Hot Memory更新延迟 <500ms（Nudge触发后）
- ✅ 数据归档规则配置完成，自动化测试通过
- ✅ 产出《数据模型文档》v1.0

### Sprint 7：Orchestrator + 异步队列 + 轻量RAG（Week 13-14）

**Sprint目标**：搭建调度中枢和事件总线，实现Agent协作骨架

**用户故事**：
1. 作为技术负责人，我想要配置Orchestrator规则引擎，以便按事件类型分发到对应Agent
2. 作为后端开发，我想要实现异步队列（Redis），以便削峰填谷处理高峰期事件
3. 作为后端开发，我想要搭建轻量RAG（关键词索引），以便Content Agent和Coach Agent检索知识

**任务分解**：
- [ ] Orchestrator规则引擎：10+事件类型定义 + 决策规则配置 + Agent分发逻辑
- [ ] 异步队列：Redis部署 + 事件网关（合并/去重）+ 规则引擎Worker（4个）+ LLM代理Worker（2个）
- [ ] 限流策略：飞书API批量写入（每5秒）/ 百炼API令牌桶（80%RPM）/ 消息推送（每秒5条）
- [ ] 轻量RAG：课程库/案例库/资源库关键词索引构建（目标命中率>70%）+ 云文档搜索API降级路径
- [ ] 冲突解决规则：预测Agent > 内容Agent（学员状态优先）

**验收标准**：
- ✅ Orchestrator可正确分发80%标准事件到对应Agent
- ✅ 异步队列可处理50并发事件无丢失
- ✅ 轻量RAG关键词索引命中率 ≥70%（PoC验证）
- ✅ 规则引擎内部延迟 <200ms，端到端 <2s（80%场景）

**风险**：
- ⚠️ 飞书API限流 → 对策：批量写入+本地缓存+异步队列已配置
- ⚠️ 轻量RAG命中率不足 → 对策：扩充关键词库或提前引入语义检索

**任务分解**：
- [ ] 诊断Agent：作品分析Prompt调优（节奏/调色/字幕/叙事4维度）+ 微场景测试生成 + 能力缺口热力图

- [ ] Prompt工程师专项：CoachAgent基础Prompt v1.0（苏格拉底式提问模板）

### 6.2 应急预案

**LLM供应商故障**：
- 自动切换至备用供应商（百炼→豆包→Claude）
- 若全部不可用，规则引擎接管关键功能（预警/推送/记录）
- 通知技术负责人手动介入

**飞书API限流**：
- 异步队列缓冲事件，避免丢失
- 批量写入替代单条操作
- 本地缓存减少API调用

**重大Bug上线后**：
- P0 Bug：立即停止相关Agent，手动修复后验证
- P1 Bug：24h内发布热修复
- P2 Bug：下个Sprint修复
