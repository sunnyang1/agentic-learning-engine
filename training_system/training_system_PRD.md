## 二、产品概述

### 2.1 产品背景

MediaStorm（影视飓风）当前团队规模约 **100人**，年产出 **130+部影片**。随着业务高速增长，团队面临严峻的人才管理挑战：

- **编导岗位年淘汰率约10%**（每年淘汰约10人），人员流动成本高
- **新人上手周期长**：剪辑师/编导新人需要2-3个月才能独立产出符合"影视飓风风格"的内容
- **知识传承依赖个人**：优秀编导的经验（如Tim的选题逻辑、剪辑节奏把控）没有系统化沉淀
- **派活匹配靠直觉**：制片安排项目时，不清楚哪位剪辑师擅长快节奏评测、哪位擅长纪录片叙事
- **晋升标准模糊**：员工看不到从"剪辑助理→剪辑师→高级剪辑师→剪辑指导"的清晰路径

### 2.2 产品定位

> **一句话描述（v1.0）**：为MediaStorm打造的"人才成长基础设施"——让新人快速上手、让老员工经验可传承、让管理者清晰看到每个人的能力地图。
>
> **一句话描述（v2.0 — Agentic 重构）**：为MediaStorm打造的**"AI 数字教练团队"**——不是让新人去系统里找课学，而是让 AI Agent 主动找到新人的能力缺口、主动推送刚好需要的内容、主动在剪辑时实时指导、主动在要放弃前介入干预。人类导师从"重复教基础"解放出来，专注"启发创意"。
>
> **核心范式转变**：
> ```
> v1.0（传统 LMS）：学员 → 打开系统 → 找课程 → 学完 → 做测评 → 等待反馈
> v2.0（Agentic）：  Agent 感知学员状态 → 自主决策 → 调用工具层 → 主动推送服务 → 实时反馈 → 动态调整
> ```

### 2.3 产品目标

| 目标维度 | 具体目标 | 衡量指标 |
|---------|---------|---------|
| **效率** | 缩短新人独立产出周期 | 从14天 → 7天（MVP目标<14天，成熟期<7天） |
| **质量** | 降低因"风格不符"导致的人才淘汰 | 编导岗位淘汰率从10% → 5% |
| **传承** | 将优秀员工的经验系统化沉淀 | 建立可复用的"影视飓风方法论" |
| **决策** | 让派活和晋升有据可依 | 100%项目派活基于能力图谱匹配 |
| **评估** | 季度 360° 评估自动化 | 评估周期从2周 → 1周，人工统计降为零 |
| **激励** | 导师带教可量化激励 | 导师积分与培训预算/调休/荣誉挂钩 |

### 2.4 产品范围

**In Scope（本期范围）**：
- 剪辑师岗位的能力图谱与培训体系（MVP首选，人数最多、标准化程度最高）
- 编导岗位的入门课程与案例库
- 新人入职学习路径与测评机制
- 导师匹配与辅导跟踪
- **季度 360° 评估系统**（飞书审批流 + 自动汇总，v3.0 新增）
- **等级认证体系**（L1-L4 四级晋升，v3.0 新增）
- **导师激励积分制**（积分自动累计 + 兑换，v3.0 新增）

**Out of Scope（本期不做）**：
- 摄影师/制片等岗位的能力图谱（二期扩展）
- **导师激励与薪酬系统直接对接**（仅兑换培训预算/调休/荣誉，v3.0 明确边界）
- 与项目管理系统/财务系统深度对接（Phase 4 远期）
- AI自动评分视频作品（远期规划）
- 360°评估中的同事匿名评价（MVP 不启用，Phase 2 补充）

## 四、需求分析

### 4.1 痛点-场景-需求映射

| 痛点（Pain） | 场景（Context） | 需求（Need） | 优先级 |
|-------------|----------------|-------------|--------|
| 新人没人教 | 入职第一周，坐在工位无所适从 | 结构化的新手引导与必修课程 | P0 |
| "影视飓风风格"模糊 | 新人剪完一版被打回"没有网感" | 可量化的风格标准与案例库 | P0 |
| 派活靠印象 | 制片人凭记忆安排人员，经常 mismatch | 可视化的能力图谱与智能推荐 | P1 |
| 带教重复劳动 | 资深员工每次教同样的基础内容 | 标准化课程库 + 导师专注高级技巧 | P1 |
| 晋升标准模糊 | 员工不知道做到什么程度可以升职 | 清晰的成长路径与晋升标准 | P1 |
| 经验随人流失 | 优秀员工离职，经验没有沉淀 | 案例拆解库与方法论沉淀 | P2 |

### 4.2 竞品分析

| 竞品 | 类型 | 优点 | 缺点 | 对MediaStorm的启示 |
|------|------|------|------|-------------------|
| **Notion** | 知识库/wiki | 灵活、美观、容易上手 | 缺乏能力评估和路径规划功能 | 可作为知识库的UI参考 |
| **Confluence** | 企业Wiki | 结构化文档、权限管理强 | 学习成本高、不够轻量 | 适合作为底层文档系统 |
| **企业微信学习平台** | LMS | 课程-考试-证书闭环 | 通用设计，不适合创意行业 | 测评机制可参考 |
| **Eagle** | 素材管理 | 视觉化、适合创意人 | 纯工具，没有培训逻辑 | 界面风格可参考 |

**结论**：MediaStorm 需要的是一个**"知识库 + 能力评估 + 成长路径"三位一体的轻量级系统**，而不是传统的LMS（学习管理系统）。

**v2.0 升级结论**：MediaStorm 需要的不是「一个系统」，而是**「一支 7×24 在线的 AI 数字教练团队」**——由 Orchestrator 统一调度，多个专业 Agent 协作，背后连接课程库、案例库、能力图谱、导师网络等工具层。学员感受不到「在使用系统」，而是感受到「有一位懂我的教练在陪伴我成长」。

### 模块 0：Agentic Learning Engine（智能学习引擎）

> **核心理念（v2.0）**：传统 LMS 假设"人主动找内容"，v1.0 Agentic Learning 引入了 5 个独立 Agent。v2.0 的升级在于：**5 个 Agent 不再独立工作，而是在 Orchestrator（调度中枢）的统一编排下，共享 Memory（学员记忆），通过 Tool Registry（工具注册表）调用工具层，形成「感知 → 决策 → 行动 → 反馈」的自主循环**。
>
> **架构升级对比**：
> ```
> v1.0（独立Agent）：学员 → 诊断Agent → 内容Agent → CoachAgent → 学员（线性串行，Agent之间不共享状态）
> v2.0（协作Agent网络）：
>         事件触发（学员行为/时间/状态变化）
>              ↓
>     Orchestrator（调度中枢）接收事件，查询 Memory
>              ↓
>     决策：哪些Agent参与？什么顺序？是否需要Human-in-the-Loop？
>              ↓
>     Agent并行/串行执行 → 调用Tool Registry中的工具 → 更新Memory
>              ↓
>     结果反馈给学员/导师/管理者
> ```
>
> **设计灵感来源**：2025-2026 年最前沿的 Multi-Agent Systems（多智能体系统）。Gartner 预测到 2027 年，约 70% 的多智能体系统将包含专业分工的 Agent。Khanmigo、Syntea、CYPHER Learning、AutoGen、LangGraph 等框架已证明：真正的 Agentic AI 不是单个 Agent 做所有事，而是多个专业 Agent 在 Orchestrator 调度下协作完成复杂任务。

#### 0.1 为什么影视飓风需要 Agentic Learning v2.0？

| v1.0 的局限 | 影视飓风的深层挑战 | v2.0 多 Agent 协作的解法 |
|------------|-------------------|------------------------|
| 5 个 Agent 各自为政，可能重复提问或给出矛盾建议 | 新人同时需要诊断、内容推送、教练答疑，信息冲突会加剧焦虑 | **Orchestrator + Memory** 统一调度，确保各 Agent 看到同一份学员状态，避免冲突 |
| Agent 只做"推荐"，无法自主执行多步骤任务 | "发现学员卡住"和"实际帮助学员"之间隔着多个系统操作 | **Agent 自主调用工具**：预测 Agent 发现风险 → Orchestrator 决策 → 内容 Agent 组装干预内容 → Coach Agent 发起对话 → 自动记录到 Memory |
| 缺乏全局视角，无法处理复杂场景 | "这个学员既需要补网感，又在做情感类项目，导师还说他最近情绪低落"——需要综合判断 | **Orchestrator 决策仲裁**：综合多个信号，决定优先级最高的干预方式 |
| 学员状态分散在各模块 | 课程进度在A表、测评成绩在B表、项目数据在C表，Agent 无法 holistic 判断 | **Memory 统一存储**：长期记忆（能力图谱）+ 短期记忆（当前状态）+ 工作记忆（会话上下文） |

##### **核心组件 1：Orchestrator（调度中枢）**

**角色定位**：整个 Agent 团队的「项目经理」，负责接收事件、查询记忆、决策调度、管理冲突、协调人机交接。

**输入（Event Sources）**：
| 事件类型 | 触发源 | 示例 |
|---------|--------|------|
| **学员行为事件** | 学习系统埋点 | 学员完成课程、在某页面停留超阈值、提交作业 |
| **工作流事件** | 飞书/PR/AE | 学员打开新项目、导出视频、提交审片 |
| **时间事件** | 定时任务 | 每周一早 9 点、每日晚 8 点、入职第 N 天 |
| **状态变化事件** | Memory 更新 | 能力等级 L1→L2、风险评级 🟢→🟡、项目完成 |
| **外部事件** | 导师/HR/管理者 | 导师标记「需要关注」、HR 发起试用期评估 |

**决策逻辑（Decision Logic）**：
```
事件进入 Orchestrator
    ↓
查询 Memory：该学员的完整状态快照
    ├─ 长期记忆：能力图谱、学习偏好、历史项目、性格标签
    ├─ 短期记忆：本周学习进度、当前项目、最近 3 次测评
    └─ 工作记忆：当前是否有进行中的对话/任务/干预
    ↓
决策引擎（规则 + 模型混合）：
    ├─ 规则层：硬编码的业务逻辑（如「连续3天未登录→必须干预」）
    ├─ 策略层：可配置的干预策略（如「优先Coach Agent vs 优先内容Agent」）
    └─ 模型层：轻量预测模型（如「该学员此刻最可能接受什么形式的帮助？」）
    ↓
输出：Agent 执行计划
    ├─ 执行哪些 Agent（单 Agent / 多 Agent 并行 / 多 Agent 串行）
    ├─ 执行顺序和依赖关系
    ├─ 是否需要 Human-in-the-Loop（人工审核/人工执行）
    └─ 预期产出和成功标准
```

**冲突解决示例**：
> **场景**：内容 Agent 想推送「网感特训课程」（因为学员网感得分 L1.5），但预测 Agent 检测到学员今天已学习 4 小时、情绪指标下降（通过飞书消息响应速度判断）。
>
> **Orchestrator 决策**：
> 1. 查询 Memory：学员今日学习时长 4h，历史平均 2.5h，已超额
> 2. 冲突仲裁：预测 Agent 的「疲劳预警」优先级 > 内容 Agent 的「内容推送」
> 3. 执行计划：
>    - 暂不推送新课程
>    - 由 Coach Agent 发送关怀消息："今天学了很多了，休息一下？我明天早上再给你安排。"
>    - 记录决策到 Memory：「2026-04-19 20:00 因疲劳预警暂停推送」
>    - 明日早 9 点定时事件重新触发内容推送

**飞书落地**：飞书应用引擎工作流（Workflow）+ 飞书多维表格（状态存储）+ 飞书机器人（消息分发）

##### **核心组件 2：Memory（学员记忆）— 参考 Hermes Agent 四层记忆架构**

> **设计灵感来源**：Hermes Agent（Nous Research, 2026）采用 **四层记忆系统** 解决 Agent 的「记忆衰减」问题：Hot Memory（始终在线）+ Cold Recall（按需检索）+ Procedural Memory（技能自学习）+ External Knowledge（外部知识）。本设计将 Hermes 的记忆范式适配到企业培训场景，解决「学员背景差异大、干预策略难以复用」的问题。
>
> **核心升级对比**：
> ```
> 原设计（三层）：长期记忆 + 短期记忆 + 工作记忆
>                 ↓ 问题：Agent 每次推理都加载全部记忆，上下文膨胀；成功干预经验无法复用
> 新设计（四层）：Hot Memory + Cold Recall + Procedural Memory + External Knowledge
>                 ↓ 解决：Hot 保稳定、Cold 按需取、Procedural 积累技能、External 共享知识
> ```

**关键设计原则（参考 Hermes Agent）**：

| 原则 | Hermes 原设计 | 本系统适配 | 解决的问题 |
|------|-------------|-----------|----------|
| **Hot/Cold 分离** | Prompt Memory 始终在上下文；Session Archive 仅显式搜索时访问 | Hot Memory（USER.md + MEMORY.md）每次推理注入；Cold Recall 按需检索 | Agent 上下文膨胀；无关历史干扰当前决策 |
| **架构决定访问** | 不是 Agent 判断「要不要搜索历史」，而是由架构强制分层 | Orchestrator 根据事件类型决定读取哪层记忆 | Agent 遗漏关键历史信息；过度检索浪费时间 |
| **Nudge 机制** | 定期提示 Agent 反思并保存记忆 | 每 30 分钟或关键事件后，触发 Hot Memory 更新 | 记忆不及时更新导致 Agent 基于过时信息决策 |
| **Skills 自改进** | 完成任务后生成 Skill Document，下次复用 | 成功干预后自动合成 Skill Doc；再次发生时读取→执行→对比→更新 | 每次干预都从 0 开始；无法积累组织级最佳实践 |
| **冻结快照** | MEMORY.md/USER.md 作为冻结快照加载，保持缓存稳定 | Hot Memory 采用「写入即存盘，下次调用生效」机制 | 频繁读写导致 LLM 前缀缓存失效，增加推理成本 |

**记忆读写流程示例**：

```
【场景】Coach Agent 准备与小夏进行苏格拉底对话

Step 1: Orchestrator 加载 Hot Memory
  → 读取 USER.md：「小夏，内向，喜欢苏格拉底式，当前目标 2 周后独立剪项目」
  → 读取 MEMORY.md：「能力缺口 Top 1：网感 L1.5 | 当前项目：科技评测（截止明天）|
     风险：🟡 | 导师：阿杰 | 上次辅导：3 天前」
  → 注入 Coach Agent 的系统提示

Step 2: Coach Agent 判断是否需要 Cold Recall
  → 「小夏上周的网感练习得分如何？」→ 调用 session_search("小夏 网感 练习 上周")
  → 返回：「上周 3 次网感练习得分：68, 75, 72」
  → 判断：「得分在提升但尚未达标，继续苏格拉底式引导，不直接给答案」

Step 3: Coach Agent 判断是否需要 Procedural Memory
  → 「有没有针对『网感不足+内向学员』的成功干预策略？」
  → 调用 skill_search("网感不足 内向学员 苏格拉底")
  → 返回 Skill Doc：《内向学员网感提升的苏格拉底引导策略》
  → 读取：「内向学员需要先建立安全感，前 3 个问题用封闭式，之后逐步开放」

Step 4: Coach Agent 执行对话
  → 综合 Hot + Cold + Procedural，生成个性化对话策略
  → 对话结束后，评估效果

Step 5: Nudge 触发 Hot Memory 更新
  → 本次对话得分：学员参与度 8/10，思考深度 7/10
  → 更新 MEMORY.md：「最近关键事件：Coach 对话，网感思考深度 7/10」
  → 写入磁盘（下次 Agent 调用时生效）

Step 6: 如干预成功，更新/生成 Skill Doc
  → 「网感 L1.5 + 内向 + 苏格拉底」干预成功
  → 查询是否已有相关 Skill Doc
  → 如有：更新成功率数据 + 可能新增分支策略
  → 如无：生成新 Skill Doc，标记为 v1.0
```

**记忆更新触发条件（Nudge 机制）**：

| 触发条件 | 动作 | 更新层级 |
|---------|------|---------|
| 每 30 分钟定时 | 触发 Agent 反思当前会话，提取关键事实 | Hot Memory |
| 关键事件（能力等级变化/风险升级/项目完成） | 立即更新相关记忆卡片 | Hot Memory + Cold Recall |
| Agent 成功完成复杂干预 | 合成或更新 Skill Doc | Procedural Memory |
| 课程/案例/资源库更新 | LLM Wiki 自动重新索引 | External Knowledge |
| 每日凌晨 3:00 | 全局记忆压缩：归档过期短期记忆，更新长期统计 | Cold Recall |

##### **核心组件 3：Tool Registry（工具注册表）**

**角色定位**：Agent 可调用的「能力菜单」，Agent 不直接操作数据库，而是通过注册表调用标准化工具。

```
TOOL REGISTRY
│
├── Content Tools（内容工具）
│   ├── search_course(keyword, difficulty, tags) → 返回课程列表
│   ├── search_case(type, style, difficulty) → 返回案例列表
│   ├── get_resource(asset_type, project_type) → 返回资源包
│   └── assemble_learning_package(user_id, goal, constraints) → 返回学习套餐
│
├── Assessment Tools（测评工具）
│   ├── generate_questions(course_id, count, types) → 返回 AI 生成题目
│   ├── submit_quiz(user_id, answers) → 返回评分 + 错题分析
│   ├── get_skill_gap(user_id, target_level) → 返回缺口分析
│   └── evaluate_work(user_id, work_url, rubric) → 返回评审报告
│
├── Communication Tools（通讯工具）
│   ├── send_message(user_id, content, channel) → 飞书消息/群消息
│   ├── schedule_meeting(host_id, attendee_id, topic) → 创建飞书会议
│   ├── create_task(assignee_id, title, deadline) → 创建飞书待办
│   └── notify_mentor(mentor_id, alert_type, context) → 通知导师
│
├── Project Tools（项目工具）
│   ├── get_project_data(user_id, project_id) → 返回项目信息
│   ├── analyze_workflow(user_id, current_step) → 返回上下文建议
│   └── get_peer_work(user_id, project_type) → 返回同类型参考作品
│
└── Analytics Tools（分析工具）
    ├── get_learning_analytics(user_id, period) → 返回学习行为报告
    ├── get_team_skills(team_id) → 返回团队能力分布
    └── predict_outcome(user_id, action_plan) → 返回预测结果
```

**设计原则**：
- **标准化接口**：每个工具有明确的输入/输出/错误处理
- **权限控制**：不同 Agent 有不同的工具调用权限（如预测 Agent 不能发送消息，需通过 Orchestrator 调度通讯 Agent）
- **调用日志**：所有工具调用记录到 Memory，确保可审计
- **失败回退**：工具调用失败时，Agent 可以重试或切换替代工具

#### 0.3 六大核心 Agent 设计（v3.0 协作模式）

> v3.0 的升级：**每个 Agent 不再是独立工作，而是在 Orchestrator 调度下，通过 Memory 共享状态，通过 Tool Registry 调用工具，形成协作网络。AI Assessment Generator 从工具升级为第六大核心 Agent。**

---

##### **Agent 1：诊断 Agent（Diagnostic Agent）**

**角色定位**：新人入职第一天的「能力 CT 扫描仪」，替代标准化笔试

**工作流**：
```
入职前 48h → 发送「作品收集链接」（过往剪辑作品/毕业设计/个人频道）
       ↓
Day 1 上午 → 分析作品：节奏感、调色水平、字幕规范、叙事逻辑
       ↓
Day 1 下午 → 3 道「微场景测试」（非选择题，而是迷你实操）：
            ├─ 测试 A：给定 30 秒素材，要求按「影视飓风风格」粗剪
            ├─ 测试 B：给定 3 个音乐选项，为指定片段选择最合适的配乐
            └─ 测试 C：给定一段文案，设计分镜节奏（文字描述即可）
       ↓
Day 1 傍晚 → 生成「能力缺口热力图」+ 「个性化新手村路径」
            例：
            ┌─────────────────────────────────────┐
            │ 小夏的能力诊断报告                  │
            │ ✅ 软件操作：L2（熟练）              │
            │ ⚠️  剪辑节奏：L1.5（需强化）          │
            │ ❌ 网感/热点敏感度：L1（入门）        │
            │ ✅ 调色基础：L2                      │
            │ ⚠️  音乐情绪匹配：L1.5               │
            │                                     │
            │ 生成路径：跳过软件基础课 →           │
            │ 直接进入「节奏训练营」+「网感特训」  │
            └─────────────────────────────────────┘
```

**与传统测评的区别**：
| 维度 | 传统入职测评 | Agentic 诊断 |
|------|------------|-------------|
| 形式 | 选择题/问答题 | 作品分析 + 微实操 |
| 结果 | 一个总分 | 多维缺口热力图 |
| 后续 | 统一分配课程 | 动态生成个性化路径 |
| 频率 | 入职一次 | 每完成一个项目自动更新 |

**飞书落地**：通过飞书应用引擎 + 飞书问卷/表单收集作品，AI 分析后写入多维表格

---

##### **Agent 2：教练 Agent（Coach Agent）**

**角色定位**：永不疲倦的「苏格拉底式导师」，不直接给答案，而是通过提问引导思考

**核心能力**：
1. **苏格拉底式对话**：
   > 新人问："这段怎么剪才有网感？"
   >
   > Coach Agent 不直接给模板，而是反问：
   > "你觉得这条视频的目标观众是谁？他们在什么场景下看？"
   > "前 3 秒如果没有钩子，观众划走的概率会增加多少？"
   > "参考案例《我花30万买了一台电影机》的前 15 秒，它用了什么手法抓住注意力？"

2. **实时纠错**：新人在飞书文档中提交剪辑脚本/分镜，Agent 自动批注：
   - "第 3 镜的停留时间 0.8 秒，对比同类爆款平均 1.5 秒，是否考虑延长？"
   - "这段 BGM 的情绪与画面氛围存在偏差，推荐参考案例库标签 #情感转折 #轻电子"

3. **引导式复盘**：项目结束后，Agent 主动发起复盘对话：
   - "这期视频的完播率比你的上一个作品高 12%，你觉得哪个决策起了作用？"
   - "导师提到你的调色偏冷，而你原本想表达温暖，这个偏差是怎么产生的？"

**与传统导师的区别**：
| 维度 | 真人导师 | Coach Agent |
|------|---------|-------------|
| 响应速度 | 按导师日程 | 秒级响应，7×24 |
| 覆盖范围 | 1 位导师带 2-3 人 | 无限并发 |
| 教学风格 | 因人而异 | 标准化苏格拉底模式 |
| 成本 | 高（占用资深员工时间） | 低（边际成本趋近于零） |
| 盲区 | 可能直接给答案 | 强制引导思考 |
| **最佳实践** | **互补**：Agent 处理 80% 基础提问，真人导师专注 20% 高阶创意指导 |

**飞书落地**：飞书智能伙伴（Feishu Copilot）+ 自建知识库 + 飞书文档批注 API

---

##### **Agent 3：内容 Agent（Content Agent）**

**角色定位**：动态「学习路径组装师」，从内容库中实时拼装最适合当前学员的「学习套餐」

**工作流**：
```
接收诊断报告 + 学员目标
       ↓
查询内容库（课程/案例/文档/模板）
       ↓
考虑约束条件：
  ├─ 学员当前等级 vs 目标等级
  ├─ 可用学习时间（全职培训 vs 边做边学）
  ├─ 正在参与的项目类型（科技评测 vs 纪录片 vs Vlog）
  ├─ 学习偏好（视频 vs 图文 vs 实操）
  └─ 内容新鲜度（优先推荐近 3 个月案例）
       ↓
组装「动态学习路径」：
  ┌──────────────────────────────────────────┐
  │ 小夏的第 2 周学习套餐（预计 6 小时）      │
  │                                          │
  │ 🎯 核心目标：提升「网感」+「节奏感」      │
  │                                          │
  │ 📹 视频课（2 节 × 15 分钟）              │
  │    ├─ 《前 3 秒定生死：钩子设计的 5 种》   │
  │    └─ 《科技评测类视频的「呼吸感」》       │
  │                                          │
  │ 📋 案例拆解（2 个）                      │
  │    ├─ 《我花30万买了一台电影机》          │
  │    │   └─ 重点：开场钩子 + B-roll 节奏    │
  │    └─ 《2024 年度数码盘点》（新增爆款）    │
  │        └─ 重点：信息密度与观众疲劳平衡     │
  │                                          │
  │ 🛠️ 微实操（3 个）                        │
  │    ├─ 练习 1：给一段无音乐素材选配 BGM     │
  │    ├─ 练习 2：给 3 个标题打分（为什么 A 比 B 好）│
  │    └─ 练习 3：30 秒素材粗剪（提交后 Agent 自动评审）│
  │                                          │
  │ 💬 苏格拉底对话（1 次，15 分钟）           │
  │    └─ 主题：如何判断一个选题有爆款潜力？   │
  └──────────────────────────────────────────┘
```

**动态调整机制**：
- 如果学员在「网感」练习中连续 3 次得分 > 85%，自动减少相关课程，增加「进阶调色」
- 如果学员在某模块停留时间超过平均值 200%，自动推送「简化版」或「1 对 1 真人辅导」预警
- 每周一早上 9 点，自动推送本周学习套餐到飞书消息

**与传统固定路径的区别**：
| 维度 | 固定学习路径 | Agentic 动态路径 |
|------|------------|-----------------|
| 生成方式 | 管理员预设 | Agent 实时组装 |
| 更新频率 | 季度更新 | 每周/每项目更新 |
| 个性化 | 按岗位分 3-5 套模板 | 千人千面 |
| 内容来源 | 仅限课程库 | 课程+案例+文档+模板+外部资源 |

---

##### **Agent 4：预测 Agent（Predictive Agent）**

**角色定位**：学员流失的「早期预警雷达」，在学习行为数据中识别风险信号

**监控指标**：
| 风险信号 | 阈值 | 自动干预动作 |
|---------|------|------------|
| 连续 3 天未登录学习系统 | 触发 | 飞书私信："小夏，遇到瓶颈了吗？我推荐了一个 5 分钟短视频，或许能帮你重启。" |
| 某模块停留时间 > 平均值 200% | 触发 | 推送「简化版指南」+ 询问"需要导师 1 对 1 吗？" |
| 测评分数连续下降 | 触发 2 次 | 自动通知导师 + HR，标记「关注名单」 |
| 作业提交质量波动大 | 触发 | 主动发起复盘对话："我注意到你最近两次作业风格差异很大，是有意尝试还是遇到了困难？" |
| 临近入职 7 天，完成率 < 60% | 触发 | 升级预警给直属领导，建议调整首周工作安排 |

**影视飓风场景的特殊价值**：
> 传统公司可以"慢慢培养"，但影视飓风的编导岗位年淘汰率 10%。如果能在第 3 天就发现"这位新人的审美风格与公司方向存在根本差异"，就可以在第 1 周就进行干预或调岗，而不是等到 3 个月试用期结束。

**飞书落地**：飞书多维表格自动化流程 + 飞书机器人消息推送

---

##### **Agent 5：实战 Agent（Practice Agent）**

**角色定位**：嵌入工作流的「即时教练」，在真实剪辑过程中提供上下文感知的指导

**触发场景与动作**：

| 工作流场景 | Agent 感知 | 即时动作 |
|-----------|-----------|---------|
| 新人在 PR 中打开「情感类纪录片」项目 | 识别项目类型标签 | 飞书侧边栏自动展开：「情感类剪辑锦囊」——推荐 3 个参考案例 + 情绪曲线模板 |
| 新人在时间轴第 2 分钟停留超过 5 分钟 | 识别可能的「卡顿点」 | 弹出："这个段落是不是在纠结节奏？参考《XXXX》的 1:45-2:10 处理方式" |
| 新人导出视频前 | 检测导出设置 | 自动核对：码率、分辨率、字幕样式是否符合品牌规范 |
| 新人提交初剪给导师 | 自动预审 | Agent 先过一遍：检查明显的技术问题（黑帧、音画不同步、品牌元素缺失），减少导师返工时间 |
| 项目数据回传（完播率/互动率） | 关联学员作品 | 自动对比该学员历史作品数据，生成「进步趋势图」 |

**与传统"先学后做"的区别**：

传统模式：`学课程 → 做作业 → 做真实项目`（脱节）

Agentic 模式：`做真实项目 → 遇到困难 → Agent 即时推送相关课程/案例 → 边做边学 → 项目完成 → 自动复盘`

> **核心理念**：学习发生在「最近发展区（Zone of Proximal Development）」——不是学完了再做，而是在「差一点就能独立完成」的任务中，Agent 提供刚好够用的支架（Scaffolding）。

#### 0.4 多 Agent 协作的端到端示例

> 以下场景展示 Orchestrator 如何调度多个 Agent，通过 Memory 共享状态，完成一个复杂的学员干预任务。

**场景：新人「小夏」入职第 5 天，在剪辑第一个实战项目时遇到困难**

```
【事件触发】14:23 — 实战 Agent 通过项目工具检测到：
  • 小夏在 PR 中打开「情感类纪录片」项目
  • 在时间轴 1:45-2:30 段落停留超过 8 分钟（阈值 5 分钟）
  • 同类型学员平均停留 3 分钟
        ↓
【Orchestrator 接收事件】14:23:05
  • 查询 Memory：
    ├─ 长期记忆：小夏「情感叙事」能力 L1.5，「节奏感」L1.5
    ├─ 短期记忆：小夏今日已学习 2.5 小时，当前项目截止日明天
    └─ 工作记忆：Coach Agent 14:00 刚与小夏完成一次对话，主题是「项目需求理解」
  • 决策引擎判断：
    ├─ 规则：「卡顿超阈值」→ 必须干预
    ├─ 策略：学员有 deadline 压力，优先推送「即时可用」的参考，而非长篇课程
    └─ 模型：小夏历史偏好显示「看案例 > 读文档」，优先推送案例
        ↓
【Orchestrator 生成执行计划】14:23:10
  Step 1（并行）：
    ├─ 实战 Agent → 推送「情感类剪辑锦囊」（案例 + 情绪曲线模板）
    └─ 内容 Agent → 查询是否有「5 分钟以内」的情感节奏微课
  Step 2（串行，依赖 Step 1 结果）：
    └─ Coach Agent → 如果 10 分钟后小夏仍在卡顿，发起关怀对话
  Step 3（条件触发）：
    └─ 预测 Agent → 监控小夏后续 30 分钟行为，如持续卡顿则升级预警给导师
        ↓
【Step 1 执行】14:23:15
  实战 Agent 调用 Tool → send_message(小夏, "情感类剪辑锦囊", 飞书侧边栏)
    内容：「检测到你在处理情感段落，推荐参考：
           ① 案例《XXXX》1:45-2:10 的情绪转折处理
           ② 情绪曲线模板（可直接下载）
           ③ 3 分钟微课《情感段落节奏：从压抑到释放》」
  内容 Agent 调用 Tool → search_course("情感节奏", max_duration="5min")
    结果：找到 2 门微课，标记为「待推送」
        ↓
  Orchestrator 更新 Memory（短期记忆）：
    「14:23 实战 Agent 已推送情感类锦囊，内容 Agent 找到 2 门微课待推送」
        ↓
【Step 2 执行】14:33:15（10 分钟后）
  Orchestrator 查询 Memory：
    「小夏 14:23-14:33 期间，打开了案例《XXXX》，观看了 2 分钟微课，
     时间轴 1:45-2:30 段落的停留时间从 8 分钟降至 2 分钟」
  → 判断：干预有效，跳过 Coach Agent 关怀对话
        ↓
  Orchestrator 更新 Memory：
    「14:33 干预成功，小夏已恢复流畅剪辑，Coach Agent 对话取消」
        ↓
【Step 3 执行】14:53:15（再过 20 分钟）
  预测 Agent 查询 Memory：
    「小夏 14:33-14:53 期间，完成了情感段落剪辑，进入下一段落，
     整体剪辑速度恢复正常」
  → 判断：无需升级预警
        ↓
  预测 Agent 更新 Memory：
    「14:53 风险解除，本次干预闭环完成」
```

**如果干预失败的分支示例**：
> 假设 Step 2 中，小夏 10 分钟后仍在卡顿：
> 1. Orchestrator 触发 Coach Agent 发起关怀对话
> 2. Coach Agent 通过对话发现：小夏不是技术问题，而是「导演给的反馈很模糊，不知道改什么」
> 3. Coach Agent 更新 Memory：「问题根因：需求沟通，非技术能力」
> 4. Orchestrator 重新决策：
>    - 推送课程《如何与导演高效沟通需求》
>    - 通知导师阿杰：「小夏遇到需求沟通问题，建议 15 分钟 1 对 1 对齐」
>    - 创建飞书待办：小夏与导演确认修改清单

#### 0.5 Agentic Learning 的技术落地（飞书生态）

| 组件 | 飞书能力支撑 | 实现难度 | 优先级 |
|------|------------|---------|--------|
| **Orchestrator** | 飞书应用引擎工作流（Workflow）+ 多维表格（状态存储）| 中 | P0 |
| **Memory** | 飞书多维表格（结构化状态）+ 飞书文档（非结构化记忆）| 中 | P0 |
| **Tool Registry** | 飞书应用引擎（API 封装）+ 多维表格（工具配置表）| 中 | P0 |
| **诊断 Agent** | 飞书问卷收集作品 + 应用引擎调用 AI 分析 + 多维表格存储结果 | 中 | P0 |
| **Coach Agent** | 飞书智能伙伴（Copilot）+ 自建知识库（飞书文档）| 低 | P0 |
| **内容 Agent** | 多维表格（内容库）+ 应用引擎（路径算法）+ 飞书消息推送 | 中 | P0 |
| **预测 Agent** | 多维表格自动化 + 飞书机器人 | 低 | P1 |
| **实战 Agent** | 需与 PR/AE 插件或飞书侧边栏集成 | 高 | P2（二期）|
| **AI Assessment Generator** | 飞书文档输入 + 智能伙伴生成 + 多维表格 Pending Pool | 中 | P1 |

#### 0.4 Agentic Learning 的评估指标

| 指标 | 传统 LMS 基准 | Agentic Learning 目标 | 测量方式 |
|------|------------|---------------------|---------|
| 新人独立产出周期 | 2 个月 | **2 周** | 从入职到首次独立交付项目 |
| 课程完成率 | 60-70% | **85%+** | 7 天新手任务完成率 |
| 导师带教时间占比 | 30% | **15%**（Agent 替代基础答疑）| 导师每周工时统计 |
| 风格不匹配淘汰率 | 10% | **5%** | 试用期淘汰原因分析 |
| 学员满意度（NPS）| — | **> 50** | 每月匿名调研 |
| 学习路径个性化覆盖率 | 0%（统一路径）| **100%** | 系统生成的路径与固定模板差异度 |

#### 0.5 关键设计原则

1. **Human-in-the-Loop（人机协同）**：Agent 不替代真人导师，而是处理 80% 重复性工作（基础答疑、路径推送、预警），让真人导师专注 20% 高价值工作（创意指导、情感支持、职业规划）。

2. **透明度与可解释性**：Agent 的每次推荐必须说明理由——"推荐这门课，因为你上周在「节奏感」测试中得分 L1.5，而你的目标岗位需要 L2.5"。学员可以随时查看自己的学习数据。

3. **隐私与安全边界**：作品分析数据仅限培训系统使用，不与绩效考核直接挂钩（避免学员"为考试而学"）。导师评价和作品数据遵循最小权限原则。

4. **渐进式信任建立**：MVP 阶段 Agent 只做"推荐"不做"决策"（如推荐课程但不强制完成），积累 3 个月数据后，逐步开放自动调整路径、自动触发预警等高级功能。

#### 0.6 测试题自动生成与审阅工作流（AI Assessment Generator）

> **设计背景**：传统培训系统的测试题需要人工编写，耗时且难以覆盖每门课程。Agentic Learning 要求「学完后立即有针对性测评」，因此测试题必须随课程自动产生。但 AI 生成的题目可能存在措辞偏差、考点偏离、难度失控等问题，必须经课程 Owner（领域专家）审阅后才能发布。

**完整工作流**：

```
课程创建 / 内容更新（变动 > 30%）
        ↓
┌─────────────────────────────────────────────────────────────┐
│  触发条件                                                   │
│  • 新课程首次发布                                           │
│  • 已有课程内容更新且变更率 > 30%（基于文本差异分析）         │
│  • Owner 手动点击「重新生成测试题」                          │
│  • 季度定期巡检：Agent 识别「热门课程但无测试题」自动补生     │
└─────────────────────────────────────────────────────────────┘
        ↓
内容 Agent 自动分析课程材料
  ├─ 视频：提取语音转文本 + 关键画面时间戳
  ├─ 图文：提取标题、列表、强调段落
  ├─ 案例：提取决策点、易错环节、关键技巧
  └─ 学习目标：匹配 Bloom 认知分类（记忆/理解/应用/分析/评价/创造）
        ↓
AI 生成测试题（每门课默认 6-12 题，按内容深度动态调整）
        ↓
进入 Pending Pool（待审池）
        ↓
飞书通知课程 Owner：「《XXX》已生成 8 道测试题，待你审阅」
        ↓
Owner 在审阅界面操作
  ├─ ✅ 通过（Approve）→ 进入 Released 状态，学员学完即测
  ├─ ✏️ 编辑后通过（Edit & Approve）→ 修改措辞/难度/选项后发布
  ├─ ❌ 拒绝（Reject）→ 说明原因，Agent 重新生成或弃用
  └─ ➕ 补充（Append）→ 保留 AI 题目，Owner 手动新增题目
        ↓
Release 后自动绑定课程 → 学员学完触发「学完即测」
```

**题型设计（适配创意行业，超越传统选择题）**：

|------|------|---------|---------|---------|
| **概念理解题** | 30% | AI 从课程关键词中生成单选/多选/判断 | 自动评分 | 版权规则、品牌规范、软件操作 |
| **场景应用题** | 25% | 给定真实工作场景，问"你会怎么处理？" | AI 辅助评分 + 导师复核 | 客户临时改需求、版权争议处理 |

### 模块 3：测评与认证（Assessment）— 能力验证工具层

> **v2.0 定位转变**：测评不再是「季度统一考试」，而是 **嵌入学习流的连续验证**。AI Assessment Generator 为每门课程自动生成测试题 → 学员学完即测 → 结果实时回写能力图谱 → 内容 Agent 自动调整后续路径。整个闭环无需人工干预。
>
> **调用接口示例**：`evaluate_learning(user_id="xia_xia_001", course_id="CS_101")` → 自动抽题 → 学员答题 → AI 评分 → 更新技能等级 → 触发路径调整

#### 3.1 入职测评 — Agent 驱动的动态诊断

#### 3.1 入职测评

**测评内容**：
- 理论测试（30%）：版权知识、品牌规范、基础概念
- 实操测试（50%）：给定素材，按标准完成剪辑
- 作品评审（20%）：提交过往作品，导师评分

**通过标准**：总分≥80分，单项不低于60分

#### 3.2 AI 生成测评题库（与模块 0.6 联动）

> 本模块是「0.6 测试题自动生成与审阅工作流」的落地执行层。

**题库生命周期**：
```
课程 Owner 创建/更新课程
        ↓
AI Assessment Generator 自动出题 → Pending Pool
        ↓
Owner 审阅 → Approve / Edit / Reject
        ↓
Released 题目进入「课程绑定题库」
        ↓
学员完成课程 → 自动抽取 3-5 题（随机+难度自适应）→ 学完即测
        ↓
答题结果 → 更新个人技能档案 → 反馈给内容 Agent 调整后续路径
```

**自适应抽题逻辑**：
- 首次学习：抽取「易:中 = 6:4」，建立信心
- 复测/补考：抽取「中:难 = 5:5」，检验深度理解
- 错题强化：自动将错题对应的知识点标记为「待强化」，推送相关微课

**测评与学习路径的闭环**：
> 小夏学完《剪辑节奏》→ 测试得分 65 分（目标 80 分）→ 系统识别「B-roll 节奏」和「情绪曲线」两个子项薄弱 → 内容 Agent 自动在下周路径中增加《B-roll 快速切换技巧》+ 案例《XXXX 的情绪转折处理》→ 小夏补强后再次测试 → 得分 85 分 → 该知识点标记为「已掌握」

#### 3.3 季度能力评估（Quarterly 360° Review）— v3.0 新增完整设计

> **需求编号**：F-24

**评估维度与权重（完整版）**：
- 自评（20%）
- 导师评价（40%）
- 同事评价（20%）— MVP 暂不启用，Phase 2 补充
- Agent 数据（20%）— 学习行为、项目数据、测评成绩等客观数据自动汇总

**MVP 简化版权重（Phase 2，Month 3-4）**：
因同事评价需匿名机制，MVP 去掉该维度，权重等比例放大：

| 维度 | 原权重 | MVP 权重 |
|------|--------|---------|
| 自评 | 20% | **25%** |
| 导师评价 | 40% | **50%** |
| Agent 数据 | 20% | **12.5%** |
| 作品数据 | 20% | **12.5%** |

**MVP 权重计算公式**：
```
final_score = self_score × 0.25 + mentor_score × 0.50 + agent_data_score × 0.125 + work_data_score × 0.125
```

**技术实现**：
- 季度初飞书待办推送 → 学员完成自评问卷 → 自动创建审批流（导师评价节点）→ Agent 数据自动汇总 → 计算总分 → 生成季度成长报告 → 飞书消息推送学员+导师+HR → 结果回写能力图谱 → 触发晋升审核

**频率**：季度（每季度第1周启动，第2周完成）

```
User（用户）
├── user_id（主键）
├── name
├── role（岗位：剪辑师/编导/摄影师/制片等）
├── level（等级：助理/初级/中级/高级/指导）
├── join_date
├── mentor_id（导师ID，外键）
└── status（在职/离职/实习）

Course（课程）
├── course_id（主键）
├── title
├── category（必修/专业必修/选修）
├── type（视频/图文/测试/直播回放）
├── duration
├── tags
├── content_url
└── last_updated

LearningRecord（学习记录）
├── record_id（主键）
├── user_id（外键）
├── course_id（外键）
├── progress（0-100%）
├── completed_at
└── score（测试分数，可选）

Skill（技能）
├── skill_id（主键）
├── skill_name
├── category（技术/审美/内容/协作）
└── description

SkillAssessment（技能评估）
├── assessment_id（主键）
├── user_id（外键）
├── skill_id（外键）
├── level（L1/L2/L3/L4）
├── assessed_by（评估人）
├── assessed_at

CaseStudy（案例拆解）
├── case_id（主键）
├── title

├── play_count
├── type（科技评测/Vlog/纪录片/广告）
├── tags
├── director
├── editor

Mentorship（导师关系）
├── mentorship_id（主键）
├── mentor_id（导师，外键→User）
├── mentee_id（学员，外键→User）
├── start_date
├── status（进行中/已结束）

QuestionBank（题库 — AI 生成 + Owner 审阅）

├── ai_generated（布尔值：是否 AI 生成）
├── ai_reference（AI 生成的考点来源：课程时间戳/文档段落）
├── status（Pending / In_Review / Approved / Rejected / Released）
├── owner_id（审阅人，外键→User）
├── owner_feedback（Owner 审阅意见）
├── created_at（AI 生成时间）
├── reviewed_at（Owner 审阅时间）
└── released_at（发布时间）

LearningAssessment（学习测评记录 — 学员答题数据）
├── assessment_id（主键）
├── user_id（外键→User）
├── course_id（外键→Course）
├── question_id（外键→QuestionBank）
├── answer（学员答案）
├── ai_score（AI 自动评分，概念理解题）
├── mentor_score（导师复核分，场景/实操题）
├── final_score（最终得分）
├── is_correct（是否正确）
├── attempt_count（尝试次数）
├── submitted_at

PendingPoolMetrics（待审池运营指标）

├── rejected_count（已拒绝）

### 10.4 关键技术挑战与解法

| 挑战 | 影响 | 解法 |
|------|------|------|
| **Agent 幻觉（Hallucination）** | 生成错误的学习内容或误导性反馈 | ① RAG（检索增强生成）：Agent 回答必须基于知识库检索 ② Human-in-the-Loop：关键决策需人工审核 ③ 置信度阈值：低置信度内容标记为「待核实」 |
| **Agent 协作冲突** | 多个 Agent 给出矛盾建议 | ① Orchestrator 统一仲裁 ② Memory 共享确保信息一致 ③ 冲突解决规则：预测 Agent > 内容 Agent（学员状态优先） |
| **实时性要求** | 实战 Agent 需要在秒级响应 | ① 规则引擎处理 80% 简单场景（<100ms）② LLM 处理 20% 复杂场景（<3s）③ 预加载常用内容到缓存 |
| **数据隐私** | 学员作品和能力数据敏感 | ① 最小权限原则：Agent 只能访问需要的 Memory ② 数据脱敏：决策日志中隐藏学员真实姓名 ③ 本地化处理：敏感数据不出境 |
| **飞书 API 限制** | 高频调用可能触发限流 | ① 批量操作替代单条操作 ② 本地缓存减少 API 调用 ③ 异步队列削峰填谷 |

### 10.5 实施复杂度评估（v2.0 Agentic 架构）

| 组件 | 复杂度 | 预估工期 | 依赖条件 | 风险等级 |
|------|--------|---------|---------|---------|
| **Tool Registry + 工具层** | 低 | 2周 | 飞书 API 已可用 | 🟢 低 |
| **Memory 结构化存储** | 低 | 1周 | 多维表格已可用 | 🟢 低 |
| **Orchestrator 规则引擎** | 中 | 3周 | 需要定义完整的事件类型和决策规则 | 🟡 中 |
| **诊断 Agent** | 中 | 3周 | 需要 LLM API + 作品分析 Prompt 调优 | 🟡 中 |
| **Coach Agent** | 中 | 3周 | 需要飞书智能伙伴 + 知识库建设 | 🟡 中 |
| **内容 Agent** | 中 | 2周 | 需要内容库建设完成 | 🟡 中 |
| **预测 Agent** | 中 | 2周 | 需要 2-4 周历史数据训练基线 | 🟡 中 |
| **AI Assessment Generator** | 中 | 2周 | 需要 LLM API + Pending Pool 工作流 | 🟡 中 |
| **实战 Agent** | 高 | 6周 | 需要 PR/AE 插件或飞书侧边栏深度集成 | 🔴 高 |
| **LLM wiki（语义知识库）** | 中 | 2周 | 由 LLM 自动索引飞书文档内容，支持语义问答。MVP 可用飞书文档原生搜索替代 | 🟡 中 |

**v3.0 MVP 定义（Month 1-2）与 Phase 2 扩展（Month 3-5）**：

```
【MVP 自动化程度分层】

Tier 1: 全自动（无需人工确认）
├─ 每日学习套餐推送（Content Agent 自动组装）
├─ Coach Agent 苏格拉底对话（全自动）
├─ 行为埋点自动记录（全自动）
├─ 能力图谱自动更新（基于规则自动计算）
└─ 预测 Agent 规则预警（全自动，仅通知导师）

Tier 2: 半自动（人工抽检，非全量确认）
├─ 诊断 Agent 作品分析 → 系统自动生成结果，导师抽检 20%
├─ 学习路径推荐 → 系统自动推送，导师仅在"异常"时介入
└─ 任务完成状态 → 系统自动标记，导师抽检 10%

Tier 3: 人工确认（必须人工审核）
├─ 能力图谱正式发布（影响晋升）
├─ 学员风险升级（🟡→🔴）
└─ 导师对 Agent 结果的异议申诉
```

**MVP 必须包含（Month 1-2）**：
  ✅ Orchestrator + Memory + Tool Registry
  ✅ 诊断 Agent（作品分析 + 微场景测试）
  ✅ 内容 Agent（基础路径组装，完成率目标 >80%）
  ✅ Coach Agent（基础苏格拉底对话）
  ✅ 预测 Agent（规则层预警：连续3天未登录/停留时间>200%/入职7天完成<60%）
  ✅ External Knowledge 轻量版（多维表格关键词索引）
  ✅ 飞书消息推送 + 飞书待办
  ✅ 能力图谱可视化（人工审核后发布）

**Phase 2 必须包含（Month 3-5）**：
  ✅ 360度评估系统（飞书审批流 + 自动汇总 + 季度成长报告）
  ✅ 等级认证体系（晋升审批流 + 能力图谱校验 + 电子徽章）
  ✅ 导师激励体系（积分自动化 + 排行榜 + 兑换审批）
  ✅ Coach Agent 深度集成 + Prompt 专项调优（4周）
  ✅ 预测 Agent 轻量模型（XGBoost/LightGBM，基于4个月数据训练）
  ✅ AI Assessment Generator + Pending Pool

**MVP 不包含**：
  ❌ AI Assessment Generator（人工出题替代）
  ❌ 实战 Agent（飞书文档替代）
  ❌ 智能派活推荐（人工查询能力图谱替代）
  ❌ 360度评估同事评价（MVP 匿名机制未就绪）
  ❌ 导师激励与薪酬系统对接（Out of Scope）

## 十一、迭代路线图（Roadmap）— v3.0 五阶段演进

> v3.0 的迭代按 **Agent 成熟度** 划分——从「辅助式 AI」到「半自主 Agent」到「自主式 Agent 网络」。v3.0 相比 v2.0 的关键变化：①增加 Phase 0 内容库建设期；②Phase 2 拆分为能力扩展（Month 3-4）和认证激励扩展（Month 4-5）；③Phase 3 拆分为 Memory 扩展（3a）和实战 Agent（3b）；④总工期从 ~28 周调整为 **~33 周**。

### Phase 0：准备期（Month -0.5 ~ Month 0）

**Phase 0a（Month -0.5）：技术熟悉期**
- Week 1：飞书基础能力上手（应用引擎 Workflow / 多维表格 API / 机器人消息推送）
- Week 2：飞书进阶能力 + PoC 验证（调用百炼 LLM / 智能伙伴配置 / 团队 Demo）
- 产出：《飞书开发规范文档》+ 可复用 Workflow 模板

**Phase 0b（Month 0）：内容库建设**
- Week 1-2：内容盘点（现有材料收集 + 缺口评估）
- Week 3-4：MVP 内容生产（品牌规范课程 + 达芬奇基础 + 网感入门 + 3 个标杆案例拆解 + 基础资源包）
- 内容生产方式：资深员工口述 → Agent 辅助整理 / 已有视频 → 飞书妙记转录 + Agent 提取知识点

---

### Phase 1：Agent 基础设施（Month 1-2）
**目标**：搭建 Agent 协作骨架，实现「辅助式 AI」——Agent 做推荐，人类做决策
**Agent 自主决策比例目标**：20%

| 里程碑 | 交付物 | 验收标准 |
|--------|--------|---------|
| **Week 1-2** | Tool Registry + 工具层 | 所有工具可通过标准化接口调用 |
| **Week 3** | Memory 两层架构 | Hot + Cold Storage 可读写 |
| **Week 4-5** | Orchestrator 规则引擎 + 异步队列 + External Knowledge 轻量版 | 支持 10+ 事件类型，决策延迟内部 <200ms / 端到端 <2s |
| **Week 6-7** | 诊断 Agent MVP | 作品分析准确率 >70%，导师认可率 >80% |
| **Week 7-8** | 预测 Agent 规则层 | 硬阈值预警上线，命中率 ≥45%（基线） |
| **Week 8** | 集成测试 + 内测 5 人 | 1 名新人完整跑通入职→诊断→学习→测评流程 |

---

### Phase 2：Agent 能力扩展（Month 3-4）
**目标**：Coach + 内容 Agent 上线，实现「半自主式 AI」
**Agent 自主决策比例目标**：50%

| 里程碑 | 交付物 | 验收标准 |
|--------|--------|---------|
| **Week 1-4** | Coach Agent + Prompt 调优（4 周专项） | Coach 可处理 80% 基础提问，思考深度评分 ≥7/10 |
| **Week 3-4** | 内容 Agent + 预测 Agent 轻量模型 | 路径完成率 >80%，XGBoost 命中率目标 ≥60% |
| **Week 5-6** | AI Assessment Generator + Pending Pool | Owner 审阅通过率 >70%，审阅耗时 <10 分钟 |
| **Week 5-6** | **360度评估系统**（v3.0 新增） | 飞书审批流 + 自动汇总 + 季度成长报告 |
| **Week 7-8** | 灰度上线（剪辑师 15 人） | 课程完成率 >80%，导师时间节省 >30% |

---

### Phase 2 扩展：认证与激励体系（Month 4-5）
**目标**：等级认证 + 导师激励上线
**Agent 自主决策比例目标**：50%（不变）

| 里程碑 | 交付物 | 验收标准 |
|--------|--------|---------|
| **Week 1-2** | **等级认证体系**（v3.0 新增） | 晋升审批流 + 能力图谱自动校验 + 电子徽章 |
| **Week 2-3** | **导师激励体系**（v3.0 新增） | 积分自动化统计 + 排行榜 + 兑换审批流 |
| **Week 3-4** | 认证/激励/评估三模块联调 | 与 Agent 闭环集成测试通过 |
| **Week 5-6** | 小范围内测 | 导师 5 人 + 学员 10 人 |

### Phase 3a：Memory 扩展（Month 5）
**目标**：Procedural Memory + External Knowledge 上线
**Agent 自主决策比例目标**：60%→70%

| 里程碑 | 交付物 | 验收标准 |
|--------|--------|---------|

**目标**：实战 Agent Phase 2 + 全量上线

| **Week 7-8** | 全量上线（分批次：剪辑师→编导→其他） | 新人独立产出周期 <14 天 |

### Phase 4：持续优化（Month 7-12）
**目标**：Agent 从「执行者」进化为「学习者」
**Agent 自主决策比例目标**：85%+

| **Month 9-10** | 跨系统集成 | 与项目管理/薪酬系统对接（远期规划） |
| **Month 11-12** | Agent 市场 + 自定义 Agent | 资深员工可创建自定义 Agent（如「Tim 的选题逻辑 Agent」） |

## 十二、风险评估与应对

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|---------|
| **资深员工不愿意分享** | 中 | 高 | 设计激励机制（积分、奖金、荣誉），让分享=收益。Agent 辅助降低分享成本：资深员工只需口述，Agent 自动整理为结构化案例 |
| **Tim没有时间配合拆解案例** | 高 | 高 | 先让资深剪辑师/编导拆解，Tim的爆款作为"特别篇」。Agent 辅助：上传原始素材和成片，Agent 自动提取时间轴、生成初版拆解框架，专家只需审核修正 |
| **新人觉得系统枯燥不用** | 低 | 中 | v2.0 从根本上解决：Agent 主动推送，学员无需「打开系统找课」。对话式交互天然比「点击课程列表」更有趣 |
| **能力评估主观性强** | 低 | 中 | Agent 持续收集多维度数据（项目数据、学习行为、测评结果），能力图谱从「季度人工评估」变为「实时自动更新」，主观性大幅降低 |
| **系统与现有飞书整合困难** | 低 | 中 | 优先使用飞书原生能力（多维表格、云文档、智能伙伴），Agent 层完全基于飞书生态构建 |
| **Agent 幻觉导致误导** | 中 | 高 | ① RAG 强制检索 ② Human-in-the-Loop 关键决策人工审核 ③ 置信度阈值：低置信度内容标记为「待核实」 ④ 建立「Agent 决策日志」审计机制 |
| **Agent 协作冲突** | 中 | 中 | Orchestrator 统一仲裁 + Memory 共享 + 冲突解决规则库。MVP 阶段限制 Agent 并发数量，逐步扩展 |
| **学员对 Agent 不信任** | 中 | 中 | 透明度设计：每次 Agent 推荐说明理由（"推荐这节课，因为你在上周测评中节奏感得分 L1.5"）。渐进式信任：MVP 阶段 Agent 只做「推荐」不做「决策」 |
| **数据隐私与合规** | 低 | 高 | 最小权限原则 + 数据脱敏 + 本地化处理。学员作品分析数据仅限培训系统，不与绩效直接挂钩 |
| **技术债务：LLM 依赖** | 中 | 高 | 设计「降级模式」：LLM 不可用时，规则引擎接管关键功能。定期评估多 LLM 供应商，避免单一供应商锁定 |
| **新增模块设计与实施脱节** | 中 | 高 | v3.0 已将 360度评估/等级认证/导师激励明确排入路线图和 MVP 边界；实施中每 2 周 Review 一次进度 |
| **预测 Agent 规则层命中率低于预期** | 中 | 中 | 设定保守基线 45-55%，明确 Phase 2 模型目标 ≥60%；规则层仅预警不决策，降低误报影响 |
| **轻量 RAG 关键词索引命中率不足** | 中 | 中 | Phase 0 PoC 验证索引命中率，目标 >70%；不达标则扩充关键词库或提前引入语义检索 |
| **Prompt 调优时间超预期** | 高 | 中 | 预留 4 周专项时间 + 配备专门 Prompt 工程师 1 人 + 三阶段 Prompt 评估机制 |
