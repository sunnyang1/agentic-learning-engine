## 🛤️ 3 条用户旅程

### 旅程 1：小夏的入职 Day 0-30
**文件**：`journeys/student-onboarding.html`

覆盖从入职第一天到第 30 天的完整成长路径：
- Day 0：收到第一条 AI 学习消息（Orchestrator 触发）
- Day 1：完成首测，Agent 组装个性化学习路径（诊断 Agent）
- Day 3：每日推送 + 苏格拉底对话（Coach Agent）
- Day 7：第一次作品提交，Agent 预审 + 导师点评（人机协作）
- Day 12：卡顿预警 + Agent 关怀（预测 Agent）
- Day 15：进入真实项目，实战 Agent 陪伴（实战 Agent）
- Day 20：学完即测，Pending Pool 题目（AI Assessment）
- Day 30：能力飞跃，银牌剪辑师候选（里程碑）

### 旅程 2：阿杰的 Agent 辅助带教
**文件**：`journeys/mentor-workflow.html`

展示导师从被动应付到主动专注高价值辅导的转变：
- Step 1：周一早上收到 Agent 预处理周报
- Step 2：周三学员提交作品，Agent 30秒技术预审 + 导师 5分钟创意点评
- Step 3：Agent 预警 → 导师快速响应
- 效果对比：v1.0 人工 vs v2.0 Agent 辅助（时间节省 85%）

### 旅程 3：Lisa 的智能派活
**文件**：`journeys/manager-assignment.html`

展示制片人如何基于能力图谱精准匹配人员：
- Step 1：输入项目类型 + 风格 + 难度
- Step 2：Agent 查询能力图谱，生成 3 位最佳匹配人员及匹配理由
- Step 3：一键确认，自动创建项目群、发送通知、更新数据库

## 📐 设计决策说明

### 为什么选择「飞书消息流」作为学员端核心入口？

根据方案设计中的「飞书原生优先」原则，学员端不需要单独打开一个培训系统，所有交互通过飞书消息完成。这符合 MediaStorm 团队已有的工作习惯（飞书是官方 Lark 案例），大幅降低学习成本。

### 为什么导师/管理端使用「桌面端仪表盘」？

导师和管理层需要查看表格、对比数据、进行批量操作，桌面端的大屏幕更适合这些任务。同时飞书多维表格和飞书应用引擎本身就是桌面端优先的产品。

### 为什么增加「智能派活」页面？

方案设计中「智能派活推荐」是一个独立的 P1 功能模块（模块 2.3）。作为管理端的核心价值场景，它需要独立的页面来展示从输入到推荐到确认的完整流程。
