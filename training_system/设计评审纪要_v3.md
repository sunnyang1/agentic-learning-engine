## 三、v4.0 新增风险识别（本轮评审核心发现）

### 3.1 🔴 P0 新风险：新增3个模块未纳入实施路线图和MVP定义

**问题描述**：
- §2.6 季度360度评估：标注为"Phase 2, Month 3-4"，但6.2路线图Month 3-4 Phase 2的具体任务列表中**完全没有**360度评估的开发任务（该阶段排了CoachAgent 4周 + 预测Agent轻量模型 + AI Assessment Generator + 灰度上线）
- §2.7 等级认证体系：在6.2路线图和6.3 MVP中**完全没有出现**
- §2.8 导师激励体系：在6.2路线图和6.3 MVP中**完全没有出现**

**影响评估**：
- 三个新增模块的设计详细度很高（§2.6: 78行 + §2.7: 49行 + §2.8: 37行 = 164行），但实际开发排期缺失
- 粗略估算新增工作量：
  - 360度评估：飞书审批流配置 + 问卷设计 + 自动汇总脚本 + 报告生成 = **2周**
  - 等级认证：晋升审批流 + 能力图谱校验 + 电子徽章 = **1周**
  - 导师激励：多表关联自动化 + 积分计算 + 排行榜 = **1-2周**
  - 三个模块联调 + 与现有Agent闭环集成 = **1-2周**
  - **总计增加约5-7周工作量**，28周工期可能延长到**33-35周**

**根因分析**：
需求跟踪矩阵评审要求"补充缺失模块的详细设计"，但在补充设计后，未同步更新实施路线图和MVP定义。属于"设计-实施"同步性缺失。

**修正建议**（v4.1纳入）：
```
Month 3-4  [Phase 2: Agent能力扩展]
            目标：Coach + 内容Agent上线，实现「半自主式AI」
            自主决策比例：50%
            
            Week 1-4  ► CoachAgent + Prompt调优（4周专项）
            Week 3-4  ► 内容Agent + 预测Agent轻量模型（XGBoost/LightGBM，基于2个月数据训练）
            Week 5-6  ► AI Assessment Generator + Pending Pool
            Week 5-6  ► 【新增】360度评估系统（飞书审批流 + 问卷 + 自动汇总）  ← 新增
            Week 7-8  ► 灰度上线（剪辑师15人）

Month 4-5  [Phase 2扩展：认证与激励体系]
            目标：等级认证 + 导师激励上线
            
            Week 1-2  ► 【新增】等级认证体系（晋升审批流 + 能力图谱校验 + 电子徽章） ← 新增
            Week 2-3  ► 【新增】导师激励体系（积分自动化 + 排行榜 + 兑换审批） ← 新增
            Week 3-4  ► 三个模块联调 + 与Agent闭环集成测试 ← 新增
            Week 5-6  ► 小范围内测（导师5人 + 学员10人） ← 新增

Month 5     [Phase 3a: Memory扩展]
            ...（原计划顺延2周）
```

**工期影响**：总工期从28周延长至**32-34周**（含Phase 0的Month -0.5和Month 0，实际约9个月）。

### 3.2 🟡 P1 新风险：360度评估MVP权重计算逻辑未明确

**问题描述**：
MVP去掉"同事评价20%"后，权重为：自评20% + 导师40% + Agent数据20% + 作品20% = 100%。

但数据模型（QuarterlyReview表）中保留了`peer_score`字段（MVP为空），`final_score`的计算公式未明确：
- 方案A：按现有权重计算，peer_score=空时按0分计入 → 实际总分上限80分，需归一化到100分
- 方案B：重新分配权重，去掉peer_score后按比例放大其他维度 → 自评25% + 导师50% + Agent数据25% + 作品25% = 125%（不对）
- 方案C：自评25% + 导师50% + Agent数据12.5% + 作品12.5% = 100%（等比例放大）

**影响评估**：
- 如果不明确，不同开发者在实现时可能采用不同逻辑，导致评分不一致
- 等比例放大（方案C）是最合理的，但文档未明确说明

**修正建议**（v4.1纳入）：
在§2.6 "MVP简化版"中明确说明：
```
MVP权重计算（同事评价字段保留为空，不参与计算）：
- 自评：20% → 25%（20/80）
- 导师评价：40% → 50%（40/80）
- Agent数据：20% → 12.5%（但建议与作品数据合并为"客观数据"40%）
- 作品数据：20% → 12.5%

更简洁方案：MVP将"Agent数据+作品数据"合并为"客观数据40%"，权重变为：
- 自评：20% → 25%
- 导师评价：40% → 50%
- 客观数据（Agent+作品）：40% → 25%
```

> **建议采用"等比例放大"方案**：自评25% + 导师50% + Agent数据12.5% + 作品12.5%，在文档中明确公式 `final_score = (self_score * 0.25 + mentor_score * 0.50 + agent_data_score * 0.125 + work_data_score * 0.125)`

### 3.3 🟡 P1 新风险：导师激励积分计算依赖多表复杂关联，飞书多维表格自动化能力存疑

**问题描述**：
§2.8导师激励的技术实现中：
- 积分存储在`MentorPoints`表，`points_history`字段为JSON格式
- 积分自动统计"通过飞书多维表格自动化，关联MentorSession/AgentDecisionLog/CaseStudy表"
- 需要计算：辅导时长→10分/小时、Agent预审通过次数→2分/次、案例拆解→20分/个、学员晋升→50分/人

**影响评估**：
- 飞书多维表格自动化支持基础的多表关联（Lookup字段），但：
  - JSON字段的聚合计算（如`points_history`中累加历史积分）可能需要外部脚本
  - 跨4张表的实时关联查询在多维表格中性能可能受限
  - 积分的"月度/季度/年度"维度统计需要复杂的筛选逻辑
- 如果飞书自动化无法支持，需要额外开发外部脚本（增加1-2周工作量）

**修正建议**（v4.1纳入）：
1. 在§2.8技术实现中增加"积分计算方案选型"：
   - 首选：飞书多维表格自动化（简单场景）
   - 备选：飞书应用引擎Workflow + 自定义脚本（复杂聚合场景）
2. 将`points_history`从JSON改为独立的`MentorPointsHistory`子表（多维表格支持子表关联），避免JSON解析

### 3.4 🟡 P1 新风险：等级认证自动触发与能力图谱人工确认的时序冲突

**问题描述**：
- §2.7认证流程："Agent自动评估 → 导师推荐 → 季度Review委员会审核 → 飞书公告"
- §6.3 MVP Tier3："能力图谱正式发布（影响晋升）"必须人工确认
- 但如果Agent自动评估触发了晋升提醒，而此时能力图谱尚未"正式发布"（导师未人工确认），会导致：
  - 能力图谱未确认 → 但等级认证流程已触发 → 数据不一致
  - 或：等级认证审核时发现能力图谱未发布 → 流程卡住

**影响评估**：
- 可能导致"误触发"晋升流程，或流程阻塞
- 在季度Review场景（§2.6结果回写能力图谱后触发晋升审核）中尤其明显

**修正建议**（v4.1纳入）：
在§2.7认证流程中增加状态校验：
```
Agent自动评估触发条件：
  ├─ 能力雷达图达标 ✓
  ├─ 项目经历数量达标 ✓
  ├─ 带教记录达标 ✓（L3/L4）
  └─ 【新增】能力图谱状态 = "已发布" ✓  ← 新增校验
     └─ 若状态 ≠ "已发布"，仅发送"预备晋升提醒"给导师，不创建晋升申请
```

### 3.5 🟡 P1 新风险：预测Agent规则层命中率基准数据缺失，可能无法达到>60%目标

**问题描述**：
- §2.4.4预测Agent规则层设计：硬编码阈值（连续3天未登录、停留时间>200%、入职7天完成<60%等）
- 行业常识：规则层（硬阈值）的典型命中率40-60%，存在大量误报（False Positive）和漏报（False Negative）
- 需求目标："有效干预率>60%"
- Phase 2轻量模型（XGBoost/LightGBM）基于2个月数据训练，数据量可能不足

**影响评估**：
- MVP阶段规则层命中率若只有45-50%，则无法达到>60%目标
- 轻量模型需要至少3-6个月的标注数据（哪些预警真正导致了学员流失/留存）才能训练出可靠模型
- 2个月数据（约40-60名学员的行为数据）对于XGBoost可能样本量不足

**修正建议**（v4.1纳入）：
1. 在§2.4.4中明确规则层的预期命中率："规则层MVP预期命中率45-55%，作为基线；Phase 2轻量模型目标命中率>60%，需至少积累4个月、100+样本的标注数据"
2. 在§6.4里程碑中增加预测Agent准确率里程碑：
   ```
   | 预测Agent规则层基线 | Month 2 末 | 规则预警触发率 | 命中率≥45%（基线） |
   | 预测Agent轻量模型 | Month 6 | XGBoost模型上线 | 命中率≥60% |
   ```

### 3.6 🟡 P1 新风险：轻量RAG调用链路延迟未拆分，Phase 1性能可能不达标

**问题描述**：
- §2.2.4轻量RAG调用链路：多维表格索引查询（<200ms）→ 命中？→ 否则降级到云文档搜索API → LLM生成
- §4.3.x性能指标已拆分两层，但轻量RAG的延迟预算未单独设定
- 云文档搜索API的响应时间未知（可能500ms-2s），降级路径无延迟保障

**影响评估**：
- 如果关键词索引命中率低（如<50%），大量查询降级到云文档搜索API，端到端延迟可能超过5s目标
- Phase 1没有轻量RAG的性能基准测试计划

**修正建议**（v4.1纳入）：
在§2.2.4或§4.3.x中增加轻量RAG延迟预算：
```
轻量RAG延迟预算（Phase 1）：
  ├─ 关键词索引命中（目标命中率>70%）：多维表格查询<200ms + LLM生成<3s = 内部<3.2s，端到端<5s
  ├─ 降级到云文档搜索（目标比例<30%）：云文档API<2s + LLM生成<3s = 内部<5s，端到端<7s（⚠️ 可能超标）
  └─ 优化方向：提高关键词索引覆盖率，减少降级比例
```
